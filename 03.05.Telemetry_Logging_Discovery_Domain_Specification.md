## 1. Purpose

The Telemetry / Logging Exposure Discovery Domain defines a deterministic, discovery-only domain responsible for producing direct-observation findings for externally reachable telemetry-, logging-, or observability-adjacent interfaces observed through approved connect-only TCP interaction on a bounded approved port set.

This domain operates in an agentless, non-exploitative manner and produces normalized `DiscoveryFinding` outputs aligned with the Core Framework schema.

This domain exists to model bounded telemetry/logging interface exposure without introducing protocol interaction, banner collection, handshake inspection, metadata extraction, log retrieval, query execution, product fingerprinting, or higher-order interpretation.

---

## 2. Architectural Boundary Statement

This domain is a discovery-only producer of direct-observation findings derived solely from connect-only TCP exposure on approved telemetry/logging-related ports, with no negotiation, no banner grabbing, no handshake inspection, no metadata extraction, and no inference.

This boundary is authoritative for all implementation, validation, and downstream consumption related to this domain.

---

## 3. Vantage Point Assumptions

* No valid credentials are supplied.
* Discovery is executed from an external or minimally privileged network position.
* No exploitation, brute force, authentication bypass, or active interrogation techniques are used.
* Observations are limited to externally observable TCP connectivity signals on approved telemetry/logging-related ports.
* No target response content is read, parsed, or interpreted.
* No upstream findings are consumed by this domain.

---

## 4. Scope

### 4.1 In Scope

* Detection of externally reachable telemetry/logging-oriented interfaces via successful connect-only TCP establishment on approved telemetry/logging ports.
* Deterministic classification of approved telemetry/logging interface exposure based on static port-to-classification association.
* Production of normalized `DiscoveryFinding` outputs using approved categories and governed evidence fields.
* Low-noise, fixed-order probing of the approved M7 telemetry/logging port set.

### 4.2 Out of Scope

* Generic network exposure detection outside the approved telemetry/logging bounded port set.
* Identity-related service exposure detection.
* Generic application/service interface discovery already governed by Application / Service Discovery.
* Authentication attempts of any kind.
* Banner grabbing.
* Response reading or parsing.
* TLS handshake initiation or inspection.
* Metadata extraction.
* Log retrieval.
* Query execution.
* Endpoint enumeration.
* Product, vendor, framework, collector, backend, or version fingerprinting.
* Inference, correlation, or posture interpretation.
* Consumption of upstream findings for meaning assignment.
* Any modification to frozen validated components without formal change control.

## 5. Relationship to Existing Domains

### 5.1 Network Discovery Domain

The Network Discovery Domain remains the authoritative producer of:

* `host_presence`
* `open_port`
* Generic bounded network exposure signals
* Its own governed `service_hint` handling within its producing contract

This Telemetry / Logging Exposure Discovery Domain does not replace, rename, or reinterpret Network Discovery behavior.

This domain MUST NOT emit `open_port` and MUST NOT duplicate generic network exposure semantics under alternate naming.

### 5.2 Identity Discovery Domain

Identity Discovery remains the authoritative producer of identity-related service exposure on approved identity ports and the authoritative owner of its governed `service_hint` contract.

This domain MUST NOT reuse Identity Discovery governed vocabulary, identity categories, or identity semantic ownership.

### 5.3 Application / Service Discovery Domain

Application / Service Discovery remains the authoritative producer of bounded application-adjacent service exposure on its approved port set and the authoritative owner of `application_service_hint`.

This domain MUST NOT reuse Application / Service Discovery governed vocabulary, categories, or semantic ownership.

### 5.4 Host Configuration Discovery Domain

Host Configuration remains an inference-only downstream consumer domain.

Telemetry / Logging Exposure Discovery is a discovery producer only and MUST NOT perform inference that belongs to Host Configuration or any future correlation capability.

## 6. Technical Approach

The Telemetry / Logging Exposure Discovery Domain extends the existing Core Framework module contract as an isolated discovery producer.

The domain performs deterministic TCP connect-only checks against an approved bounded telemetry/logging port list. Findings are produced only when a TCP connection is successfully established. No application-layer data is transmitted, no socket data is read, no protocol negotiation occurs, and no response-derived interpretation is performed.

All classification in this domain is static and deterministic. Classification is derived solely from approved destination-port association and successful TCP connection establishment.

As a producing discovery domain, this domain MUST remain discovery-only. It MUST NOT:

* consume upstream findings
* infer host posture
* correlate findings across domains
* encode hidden inference in evidence fields
* depend on downstream interpretation to define its output meaning

All interactions MUST use standard operating system socket mechanisms, bounded static timeouts, fixed execution order, and structured exception handling consistent with approved project constraints.

---

## 7. Runtime Position and Execution Ordering

This domain is a discovery producer and is expected to execute after existing discovery producers that precede it in the approved milestone sequence and before any authorized inference consumer that follows discovery producers.

For initial M7 architectural planning, the expected deterministic runtime ordering is:

1. Network Discovery Domain
2. Identity Discovery Domain
3. Application / Service Discovery Domain
4. Telemetry / Logging Exposure Discovery Domain
5. Host Configuration Discovery Domain

This ordering assumption exists to preserve architectural clarity and does not authorize this domain to consume prior findings.

This domain MUST use only local execution context and approved local configuration.

## 8. Approved Initial Port Scope

The initial approved Telemetry / Logging Exposure Discovery M7 scope is limited to the following TCP ports:

* TCP/514 — `SYSLOG`
* TCP/6514 — `SYSLOG-TLS`
* TCP/12201 — `GELF`
* TCP/24224 — `LOG-FORWARD`
* TCP/4317 — `OTLP-GRPC`
* TCP/4318 — `OTLP-HTTP`

No additional ports are authorized for this domain unless this specification is updated through controlled change.

### 8.1 Rationale for Inclusion

These ports are approved as telemetry/logging-adjacent interface ports suitable for bounded connect-only exposure observation without identity-domain overlap and without collapsing into generic application/service interface semantics.

### 8.2 Rationale for Exclusion

The following are excluded from initial M7 Telemetry / Logging Exposure Discovery scope:

* Identity-related ports already governed by Identity Discovery
* Application/service interface ports already governed by Application / Service Discovery
* Generic network ports not clearly telemetry/logging-interface focused
* Dashboard, storage, search, or admin ports whose meaning would require broader semantic interpretation
* Any port requiring product-specific, response-derived, or metadata-derived interpretation not yet formally governed

---

## 9. Probe Design Principles

All telemetry/logging probes MUST adhere to the following constraints:

* **Connect-only interaction:** Probes are limited strictly to TCP connection establishment.
* **No protocol interaction:** No syslog messages, OTLP payloads, forward-protocol exchanges, GELF messages, HTTP requests, or protocol-specific data are permitted.
* **No credential transmission:** The module must never send usernames, passwords, tokens, headers, certificates, or authentication material.
* **No response handling:** No socket reads, response parsing, handshake inspection, or metadata extraction are permitted.
* **Low-noise operation:** Timeouts and retry behavior must minimize network noise.
* **Deterministic behavior:** Identical inputs and environmental conditions must yield identical outputs.
* **Failure isolation:** Probe failures must not interrupt orchestrator execution.

## 10. Governed Classification Field

### 10.1 Field Name

`telemetry_logging_hint`

### 10.2 Definition

`telemetry_logging_hint` represents a deterministic, producer-owned classification of the approved telemetry/logging interface associated with a successfully established TCP connection on a Telemetry / Logging Exposure Discovery approved port.

It is derived strictly from:

* The destination port
* The fact that a TCP connection was successfully established
* The approved static port-to-classification mapping defined by this specification

No protocol interaction, negotiation, validation, response parsing, or metadata inspection is performed.

### 10.3 Constraints

`telemetry_logging_hint` MUST:

* Be derived solely from connect-only TCP success signals on this domain’s approved port set
* Be deterministic and consistent across executions
* Use a controlled, predefined vocabulary
* Represent classification only

`telemetry_logging_hint` MUST NOT:

* Reuse or alias `service_hint`
* Reuse or alias `application_service_hint`
* Encode reachability semantics beyond the producing contract
* Imply protocol correctness, successful ingestion, collector health, backend availability, queue state, log retention state, or service health
* Imply authentication state, TLS state, certificate validity, or response behavior
* Include inferred host roles, monitoring posture, observability maturity, or downstream platform meaning
* Be dynamically generated or environment-dependent
* Be derived from response content, banners, handshakes, headers, or metadata

### 10.4 Allowed Values (Telemetry / Logging Domain)

The following values are the authoritative controlled vocabulary for initial M7 Telemetry / Logging Exposure Discovery:

* `"SYSLOG"`
* `"SYSLOG-TLS"`
* `"GELF"`
* `"LOG-FORWARD"`
* `"OTLP-GRPC"`
* `"OTLP-HTTP"`

No additional values are permitted without controlled specification update.

### 10.5 Cross-Domain Contract Requirements

This domain is the authoritative producing owner of `telemetry_logging_hint`.

Producing logic MUST:

* Emit only approved values
* Treat `telemetry_logging_hint` as a governed evidence field
* Avoid hidden inference, role claims, posture claims, product identification, or downstream-layer meaning injection

Consuming domains MUST treat `telemetry_logging_hint` as:

* Read-only
* Authoritative
* Exact-match only
* Non-derivable outside approved consuming specifications

No domain may normalize, alias, reinterpret, extend, or silently remap the meaning of `telemetry_logging_hint`.

### 10.6 Determinism Requirements

For identical inputs:

* `telemetry_logging_hint` MUST always resolve to the same value
* Mapping between approved port and `telemetry_logging_hint` MUST remain fixed
* No conditional or environment-based variation is allowed

## 11. Finding Model

### 11.1 Output Domain

All findings generated by this domain MUST use:

* `domain = "telemetry_logging"`

### 11.2 Approved Output Category

The following `DiscoveryFinding.category` value is authorized for initial M7 implementation:

* `telemetry_logging_exposed`

No alternate category names, synonyms, or additional categories are authorized without controlled update.

### 11.3 Category Determination Constraints

* Findings are produced only upon successful connect-only TCP establishment on an approved telemetry/logging port.
* Category assignment MUST be deterministic.
* Category values MUST NOT be response-derived or dynamically generated.
* This domain MUST NOT emit generic `open_port`.

---

## 12. Evidence Collection Strategy

A probe may generate a finding when the following condition is observed:

* Successful TCP connection indicating telemetry/logging interface exposure on an approved telemetry/logging port.

Evidence must remain strictly observational and derived from TCP connectivity only.

Permitted evidence elements for positive findings include:

* `method`
* `port`
* `telemetry_logging_hint`
* `timeout_s`
* Optional controlled explanatory note consistent with this specification

Evidence MUST NOT include:

* response content
* banners
* headers
* handshake artifacts
* metadata
* certificate details
* vendor or product identification
* inferred posture or role semantics
* collector configuration
* log content
* ingestion success state
* backend naming or topology

---

## 13. Timeout and Safety Controls

To maintain enterprise-safe behavior:

* Default socket timeout must remain low (≤ 1–2 seconds).
* No aggressive retry loops are permitted.
* Parallel probing must be avoided unless explicitly authorized in a future controlled update.
* All socket operations must be wrapped in structured exception handling.
* No payload transmission is permitted after connection establishment.
* Connection must terminate immediately once success or failure is determined.

---

## 14. Responsibility Boundary Enforcement

### 14.1 What Network Discovery Owns

Network Discovery owns:

* host presence observation
* generic TCP exposure observation
* `open_port`
* generic network-domain port/service classification within its own producing contract

### 14.2 What Identity Discovery Owns

Identity Discovery owns:

* direct-observation findings for successful connect-only exposure on approved identity-domain ports
* `identity_service_exposed`
* governed `service_hint` within the identity producing contract

### 14.3 What Application / Service Discovery Owns

Application / Service Discovery owns:

* direct-observation findings for successful connect-only exposure on its approved bounded application/service port set
* `application_service_exposed`
* `application_service_hint` as its governed producer-owned classification field

### 14.4 What Telemetry / Logging Exposure Discovery Owns

Telemetry / Logging Exposure Discovery owns:

* direct-observation findings for successful connect-only exposure on its approved bounded telemetry/logging port set
* `telemetry_logging_exposed`
* `telemetry_logging_hint` as its governed producer-owned classification field

### 14.5 Overlap Prevention Rules

To prevent semantic duplication:

* This domain MUST NOT emit `open_port`
* This domain MUST NOT reuse `service_hint`
* This domain MUST NOT reuse `application_service_hint`
* This domain MUST NOT classify identity-related ports
* This domain MUST NOT classify application/service ports already reserved to Application / Service Discovery
* This domain MUST remain limited to its approved bounded scope
* This domain MUST NOT reinterpret Network, Identity, or Application / Service outputs
* This domain MUST NOT depend on Network, Identity, or Application / Service findings to suppress or produce its own outputs

## 18. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Semantic duplication with Network Discovery | High | Enforce separate category, bounded port set, and producer-owned governed field |
| Semantic duplication with Application / Service Discovery | High | Reserve distinct port set and separate governed vocabulary |
| Hidden coupling through reused vocabulary | High | Prohibit reuse of `service_hint` and `application_service_hint`; define `telemetry_logging_hint` as separate governed contract |
| Unauthorized protocol interaction | High | Enforce 01.07 connect-only methodology and hard prohibitions |
| Scope creep into inference, enumeration, or product fingerprinting | High | Preserve discovery-only boundary and forbid metadata extraction and query behavior |
| Non-deterministic behavior | Medium | Enforce fixed probe list, fixed order, static mapping, fixed timeout |
| Architectural drift into generic monitoring dashboards or data platforms | Medium | Preserve bounded initial port list and explicit exclusion rules |
