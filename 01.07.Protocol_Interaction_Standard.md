## 1. Purpose

This document defines the **Protocol Interaction Standard** for the Enterprise Security Discovery Orchestration Framework.

It establishes **mandatory technical control constraints** governing how all discovery components interact with target systems.

The purpose of this standard is to:

- Enforce **agentless operation**
- Enforce **non-exploitative behavior**
- Define and constrain **connect-only interaction methodology**
- Prevent protocol-level drift across all discovery domains
- Ensure deterministic, auditable, and non-intrusive system behavior

This document is a **control artifact (01.xx)** and applies across the full SDLC lifecycle.

## 4. Core Principle: Connect-Only Interaction

### 4.1 Definition

Connect-only interaction is defined as:

> A single, unauthenticated, non-payload-bearing TCP connection attempt to a target port, used solely to determine reachability.

---

### 4.2 Required Characteristics

All compliant interaction MUST:

- Perform a single TCP connection attempt
- Use standard operating system socket mechanisms
- Avoid transmission of any application-layer data
- Avoid protocol negotiation or session establishment
- Terminate immediately after connection result is determined

---

## 5. Allowed Behaviors

The following behaviors are explicitly permitted:

### 5.1 Network Interaction

- TCP socket connection attempt (`connect`)
- Determination of connection success or failure
- Deterministic timeout handling

---

### 5.2 Evidence Collection

Only the following data may be recorded:

- Target identifier
- Port number
- Connection result (success / failure / timeout)
- Static timeout value
- Predefined service classification (static mapping only)

---

### 5.3 Deterministic Execution

All discovery behavior MUST be:

- Deterministic
- Repeatable
- Non-adaptive

This includes:

- Fixed probe lists
- Fixed execution order
- Fixed timeout values

---

## 6. Forbidden Behaviors (Hard Prohibitions)

The following actions are strictly prohibited:

---

### 6.1 Protocol Interaction

- Sending any payload after connection establishment
- Reading or parsing any response data
- Banner grabbing of any kind
- TLS handshake initiation or inspection
- Protocol negotiation (e.g., SMB, LDAP, RPC, WinRM, RDP)

---

### 6.2 Application-Layer Interaction

- HTTP requests (GET, POST, HEAD, etc.)
- LDAP bind operations
- SMB session setup
- RPC endpoint interaction
- Kerberos ticket requests
- Any protocol-specific request or exchange

---

### 6.3 Authentication and Identity Interaction

- Use of credentials (explicit or implicit)
- Anonymous or null-session interaction
- Token or ticket acquisition

---

### 6.4 Enumeration and Data Retrieval

- Service enumeration
- Endpoint enumeration
- Metadata extraction
- Directory queries
- Any data retrieval from target systems

---

### 6.5 State-Altering or Observable Interaction

- Any action that alters target system state
- Any sustained session interaction beyond connection attempt

---

## 7. Boundary Clarifications

---

### 7.1 TCP Connect vs SYN Scan

- Full TCP connect: **Allowed**
- Raw SYN (half-open) scanning: **Forbidden**

---

### 7.2 TLS Handshake

- Initiation of TLS handshake: **Forbidden**

---

### 7.3 Connection Lifecycle

- Immediate connection termination after result: **Required**
- Reading from socket buffers: **Forbidden**

---

### 7.4 Timeout Handling

Timeout behavior MUST be:

- Static
- Deterministic
- Non-adaptive

---

### 7.5 Error Interpretation

Only the following outcomes may be used:

- Connection successful
- Connection refused
- Connection timed out

No additional interpretation is permitted.

---

### 7.6 Service Identification

Service classification MUST be:

- Derived from static port-to-service mapping
- Independent of target response

Dynamic identification is prohibited.

---

### 7.7 Controlled Service Classification Vocabulary

Where service classification is recorded as part of discovery evidence (for example, `service_hint`), the classification value MUST:

- Be derived from an approved static port-to-service mapping
- Be independent of target response
- Use the canonical vocabulary defined by the authoritative producing domain specification
- Remain deterministic for identical input conditions

Such classification values MUST NOT:

- Encode reachability semantics
- Imply protocol negotiation, authentication, enumeration, or protocol correctness
- Be dynamically generated from banners, payloads, or response content
- Be normalized heuristically by consuming domains

Any change to a governed service classification value requires:

- Controlled update to the authoritative producing specification
- Cross-domain impact review
- Regression validation
- Test artifact alignment

## 8. Deterministic Enforcement Requirements

All implementations MUST:

- Produce identical outputs for identical inputs
- Avoid adaptive logic or conditional probing
- Avoid behavior based on response characteristics
- Maintain fixed execution order

---

## 9. Enforcement Model

---

### 9.1 Code-Level Enforcement

All implementations MUST:

- Use approved connection mechanisms only
- Avoid all data transmission beyond connection attempt
- Avoid all data reception or parsing

---

### 9.2 Review-Level Enforcement

All code reviews MUST verify:

- No protocol libraries are used for interaction
- No response parsing exists
- No payload construction exists

---

### 9.3 Test Enforcement

Validation MUST confirm:

- No data retrieval occurs
- No protocol negotiation occurs
- Only connection success/failure is evaluated
- Outputs are deterministic across repeated runs

---

### 9.4 Architectural Enforcement

Discovery domains MUST NOT:

- Implement protocol logic
- Interpret protocol responses
- Perform multi-step interactions

All interpretation MUST occur in inference domains only.

---

## 10. Implementation Independence Clause

This standard defines permitted and prohibited interaction patterns independently of specific implementation details.

References to constructs such as:

- TCP connection behavior
- Service classification (e.g., service_hint)

are illustrative of compliant patterns observed in existing implementations and do not mandate specific implementation mechanisms.

All implementations MUST conform to the interaction constraints defined in this document regardless of how such attributes are derived or represented.
