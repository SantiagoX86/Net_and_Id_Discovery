## 1. Purpose

This document defines the **Domain Isolation and Layer Responsibility Standard** for the Enterprise Security Discovery Orchestration Framework.

It establishes mandatory architectural control requirements governing:

- Separation of discovery, inference, correlation, control, orchestration, and reporting responsibilities
- Producer / consumer handling of governed findings
- Prohibition of cross-domain logic leakage
- Protection against hidden coupling between producing and consuming domains
- Prevention of business-logic drift into orchestration, control, or output layers
- Safe architectural expansion for future domains, including M7 and beyond

This document is a **01.xx control artifact** and applies across the full SDLC lifecycle.

---

## 2. Scope

This standard applies to:

- All current and future discovery domains
- All inference and correlation capabilities
- The Core Framework / orchestration layer
- The CLI / control layer
- The Output & Reporting layer
- All governed findings contracts and evidence handling
- All implementation, review, validation, and closeout activities

This includes, but is not limited to:

- Network Discovery Domain
- Identity Discovery Domain
- Host Configuration Discovery Domain
- Application / Service Discovery Domain
- Telemetry / Logging Exposure Discovery Domain
- Future inference and correlation capabilities

## 4. Core Architectural Principle

The framework SHALL maintain **strict separation of responsibilities** across domains and layers.

No component may exceed its authorized responsibility boundary.

No component may reinterpret another component’s governed outputs except where explicitly authorized by an approved consuming-layer specification.

All cross-domain meaning derivation SHALL occur only in authorized inference or correlation capabilities.

---

## 5. Layer Responsibility Model

### 5.1 Discovery Layer

Discovery domains are responsible only for **direct observation** obtained through their approved interaction model.

Discovery domains MUST:

- Perform only their domain-approved observation activities
- Produce governed findings using the normalized `DiscoveryFinding` schema
- Emit direct observations only
- Remain independent of downstream interpretation logic
- Preserve deterministic execution and evidence structure

Discovery domains MUST NOT:

- Perform inference reserved for inference layers
- Perform cross-domain correlation
- Reinterpret upstream findings into higher-order conclusions
- Mutate, normalize, relabel, suppress, or extend upstream evidence
- Depend on consuming-layer logic to define their own output meaning

---

### 5.2 Inference Layer

Inference capabilities are responsible for **derived conclusions** based solely on previously produced governed findings.

Inference domains MUST:

- Consume upstream findings as read-only, append-only, contract-bound input
- Apply deterministic, specification-defined logic only
- Produce new derived findings without modifying upstream findings
- Preserve traceability from derived outputs to source findings

Inference domains MUST NOT:

- Perform direct network interaction unless separately and explicitly authorized as a discovery domain
- Modify, overwrite, reorder, or reinterpret source evidence in place
- Introduce hidden dependencies on non-governed upstream behavior

---

### 5.3 Correlation Layer

Correlation capabilities are responsible for **cross-domain relationship analysis** across previously produced findings.

Correlation capabilities MUST:

- Operate only on existing governed findings
- Preserve deterministic processing and evidence traceability
- Produce new findings or output artifacts without mutating source findings

Correlation capabilities MUST NOT:

- Perform discovery activity
- Modify upstream findings
- Reclassify source findings in place

---

### 5.4 Core Framework / Orchestration Layer

The Core Framework is responsible only for **module orchestration, sequencing, aggregation, and failure isolation**.

The Core Framework MUST:

- Execute modules in deterministic order
- Pass authorized inputs through approved contracts only
- Aggregate outputs without reinterpretation
- Preserve structured error isolation

The Core Framework MUST NOT:

- Perform discovery logic
- Perform inference logic
- Perform correlation logic
- Normalize, reinterpret, suppress, or mutate domain findings
- Contain domain-specific business rules

---

### 5.5 Control Layer

The Control Layer is responsible only for **controlled execution input handling and invocation**.

The Control Layer MUST:

- Accept and validate approved operator inputs
- Construct authorized execution context
- Invoke the orchestration framework
- Remain free of domain-specific business logic

The Control Layer MUST NOT:

- Perform discovery logic
- Perform inference or correlation logic
- Select or reinterpret findings based on domain meaning
- Introduce execution branching tied to domain conclusions unless formally approved by design

---

### 5.6 Output / Reporting Layer

The Output / Reporting Layer is responsible only for **presentation and serialization** of governed outputs.

The Output Layer MUST:

- Serialize findings and events deterministically
- Present outputs in approved formats
- Preserve evidence integrity and traceability

The Output Layer MUST NOT:

- Perform discovery logic
- Perform inference logic
- Perform correlation logic
- Alter the semantic meaning of findings
- Mutate evidence or category values during presentation

---

## 6. Discovery Domain Isolation Requirements

All discovery domains MUST operate as isolated producers of governed findings.

### 6.1 Isolation Requirements

A discovery domain MUST:

- Use only its approved inputs and local execution context
- Derive findings from its own bounded observation model
- Remain independent from consuming-layer interpretation
- Preserve deterministic ordering and output behavior

### 6.2 Prohibited Discovery Behavior

A discovery domain MUST NOT:

- Infer host posture from combinations of upstream findings
- Correlate findings across multiple domains
- Silence or suppress valid findings because of downstream interpretation expectations
- Consume upstream findings to perform higher-order meaning assignment
- Depend on future-domain logic or correlation behavior

### 6.3 Future Discovery Domains

All future discovery domains, including but not limited to:

- Application / Service Discovery
- Telemetry / Logging Exposure Discovery

MUST remain discovery-only unless a separately approved inference or correlation capability is defined.

No future discovery domain may require an architectural exception to this standard in order to be implemented.

---

## 7. Producer / Consumer Separation Requirements

### 7.1 Producing Domains

Producing domains MUST:

- Emit findings only through approved schemas and categories
- Emit evidence only from their authorized interaction model
- Use governed vocabularies where required
- Preserve deterministic output contracts

Producing domains MUST NOT:

- Emit findings whose meaning depends on downstream reinterpretation
- Encode unapproved inference inside evidence fields
- Quietly change governed vocabulary without specification and validation updates

### 7.2 Consuming Domains

Consuming domains MUST treat upstream findings as:

- Read-only
- Append-only
- Contract-bound
- Authoritative within their governed meaning

Consuming domains MUST NOT:

- Modify upstream findings in place
- Normalize, alias, reinterpret, or extend upstream evidence unless explicitly authorized by an approved consuming specification
- Depend on undocumented upstream behavior
- Treat non-governed incidental implementation details as contractual inputs

---

## 8. Governed Evidence Handling

All evidence fields used across domains SHALL be treated as governed contract elements when defined by authoritative specifications.

Where an evidence field is declared governed, including but not limited to `service_hint`, consuming components MUST:

- Use exact-match, specification-bound handling only
- Treat the value as authoritative and read-only
- Avoid heuristic interpretation, aliasing, or synonym expansion

Evidence fields MUST NOT be used to smuggle:

- Hidden inference
- Layer-crossing semantics
- Unauthorized reachability interpretations
- Role or posture conclusions not defined by the producing contract

---

## 9. Hidden Coupling Prohibition

Hidden coupling is prohibited.

Hidden coupling includes, but is not limited to:

- Dependence on undocumented finding order beyond approved deterministic contracts
- Dependence on undocumented evidence keys or incidental implementation details
- Use of upstream findings to suppress required discovery behavior
- Assumptions that a producing domain will silently normalize for a consumer
- Semantic dependence on output formatting behavior rather than governed finding content

Any required dependency between producer and consumer components MUST be:

- Explicitly documented
- Contract-bound
- Deterministic
- Traceable to governing artifacts

---

## 10. Deterministic Enforcement Requirements

All layers and domains governed by this standard MUST:

- Produce identical outputs for identical inputs
- Maintain fixed execution ordering where sequencing is defined
- Preserve consistent governed vocabulary usage
- Avoid adaptive, heuristic, or runtime-dependent interpretation logic

No component may use ambiguous or context-sensitive behavior to cross responsibility boundaries.

---

## 11. Architectural Review and Approval Requirements

Prior to approval of any new domain or architectural enhancement, review MUST confirm:

- Discovery responsibilities remain discovery-only
- Inference responsibilities remain inference-only
- Correlation responsibilities remain correlation-only
- Core Framework remains orchestration-only
- CLI remains control-only
- Output layer remains presentation-only
- Producer / consumer contracts are explicit and governed
- No hidden coupling has been introduced

If any violation is identified, implementation MUST NOT proceed until remediation is completed.

---

## 12. Test and Validation Enforcement

Validation for any new or modified domain MUST confirm, as applicable:

- No cross-domain logic leakage
- No inference behavior in discovery domains
- No business logic in orchestration, CLI, or reporting layers
- No mutation or reinterpretation of upstream findings outside approved contracts
- Deterministic behavior preserved across repeated runs
- Governing documents remain aligned with implementation behavior

Where a capability consumes upstream findings, tests MUST validate read-only, append-only, contract-bound handling.
