# 1. PURPOSE

This document defines the **target-state architecture** for the Enterprise Security Discovery Orchestration Framework.

The system is designed to perform **agentless, non-exploitative discovery and posture inference** of enterprise environments from an external, low-privilege vantage point.

This document represents a **static architectural baseline** and is not intended to evolve with milestone implementation.

---

# 2. ARCHITECTURAL PRINCIPLES

The system is governed by the following principles:

* **Agentless Operation:** No software is deployed on target systems
* **Non-Exploitative Behavior:** No vulnerability exploitation or intrusive actions
* **Connectivity-Only Discovery:** Discovery domains rely strictly on observable network signals
* **Inference-Only Analysis:** Higher-order domains derive conclusions from existing findings only
* **No Authentication:** No credential use or identity assertion
* **No Enumeration or Negotiation:** No protocol-level interaction beyond connection attempts
* **Deterministic Execution:** All operations produce consistent, repeatable results
* **Strict Domain Isolation:** Domains operate independently with no direct coupling
* **Schema-Normalized Output:** All findings conform to a unified structure

---

# 3. SYSTEM ARCHITECTURE OVERVIEW

The system is composed of four primary layers:

1. **Control Layer (CLI)**
2. **Orchestration Layer**
3. **Discovery & Inference Domains**
4. **Output & Reporting Layer**

---

# 4. CORE FRAMEWORK COMPONENTS

## 4.1 DiscoveryContext

Provides runtime execution context including:

* Target definition
* Execution metadata
* Shared configuration

## 4.2 DiscoveryModule

Abstract base class for all domains:

* Enforces standardized execution interface
* Guarantees deterministic behavior

## 4.3 DiscoveryFinding

Normalized output schema containing:

* Domain
* Category
* Target
* Evidence
* Confidence
* Timestamp

## 4.4 DiscoveryOrchestrator

Responsible for:

* Sequencing domain execution
* Maintaining deterministic ordering
* Aggregating findings across domains

# 5. DOMAIN ARCHITECTURE

Domains are divided into two categories:

---

## 5.1 Discovery Domains (Active – Connectivity-Based)

These domains generate findings through **direct network observation**.

### 5.1.1 Network Discovery Domain

Purpose:

* Identify host presence and basic network accessibility

Behavior:

* ICMP and TCP connectivity checks
* Port reachability validation

Constraints:

* No service interaction beyond connection attempts

---

### 5.1.2 Identity Discovery Domain

Purpose:

* Identify exposed identity-related services

Behavior:

* TCP connect-only probes for:

  * SMB (445)
  * NetBIOS (139)
  * WinRM (5985 / 5986)
  * RPC (135)
  * LDAP (389 / 636)
  * Kerberos (88)
  * RDP (3389)

Constraints:

* No authentication
* No protocol negotiation
* No enumeration

---

## 5.2 Inference Domains (Passive – Analysis-Based)

These domains **do not interact with the network**.

They consume findings produced by Discovery Domains and generate higher-level insights.

---

### 5.2.1 Host Configuration Discovery Domain

(M6 – Posture Inference)

Purpose:

* Infer host security posture based on externally observable signals

Inputs:

* Findings from Network and Identity domains

Behavior:

* Analyze combinations of exposed services
* Identify configuration patterns
* Infer potential hardening gaps

Constraints:

* No network interaction
* No direct dependency on other domains (input via orchestrator only)
* No modification of upstream findings

Outputs:

* Derived findings in `host_config` domain

---

### 5.2.2 Future Inference Domains (Extensible)

The architecture supports additional inference domains such as:

* Exposure Risk Scoring
* Attack Surface Classification
* Environmental Fingerprinting

All future domains must adhere to:

* Input-only data consumption
* Deterministic processing
* Schema-compliant output

---

# 6. DATA FLOW MODEL

1. Target provided via Control Layer
2. Orchestrator initializes execution context
3. Discovery Domains execute sequentially
4. Findings are collected and normalized
5. Inference Domains process aggregated findings
6. Final results passed to Output Layer

---

# 7. DOMAIN ISOLATION MODEL

* Domains **cannot call each other directly**
* All communication occurs through:

  * Orchestrator-managed findings aggregation
* No shared mutable state between domains

---

# 8. DETERMINISTIC EXECUTION MODEL

The system enforces determinism through:

* Fixed domain execution order
* Fixed probe ordering within domains
* No concurrency or parallel execution
* Stable data structures for aggregation

---

# 9. SECURITY MODEL

The system enforces:

* No credential usage
* No authentication attempts
* No protocol negotiation
* No service interaction beyond TCP connect
* No exploitation or vulnerability testing

---

# 10. OUTPUT ARCHITECTURE

All findings are:

* Structured using DiscoveryFinding schema
* Exportable as:

  * JSON (machine-readable)
  * Markdown (human-readable reports)

---

# 11. EXTENSIBILITY MODEL

The architecture supports extension through:

* Additional domains implementing DiscoveryModule
* No modification required to existing domains
* Controlled integration via orchestrator

---

# 12. NON-FUNCTIONAL REQUIREMENTS

* Deterministic behavior
* High reliability and repeatability
* Minimal system resource usage
* Clear auditability of outputs

---

# 13. ARCHITECTURAL CONSTRAINTS

* No modification of existing domain logic when adding new domains
* No coupling between domains
* No introduction of non-deterministic behavior
* No deviation from agentless, non-exploitative model
