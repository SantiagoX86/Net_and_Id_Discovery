## 1. Purpose

The Host Configuration Discovery Domain defines a deterministic, inference-only domain responsible for deriving host-level security posture insights based solely on previously observed discovery findings.

This domain operates without direct interaction with target systems and produces normalized, auditable findings aligned with the Core Framework schema.

---

## 2. Vantage Point Assumptions

* No valid credentials are supplied.
* No direct network interaction is performed by this domain.
* All inputs are derived from prior discovery domains through orchestrator-provided governed findings.
* Under the current approved specification version, this domain is authorized to consume findings from:
  * Network Discovery Domain
  * Identity Discovery Domain
  * Application / Service Discovery Domain
  * Telemetry / Logging Exposure Discovery Domain
* No modification of target systems or upstream findings occurs.
* Observations are limited to deterministic inference from externally observable signals.

---

## 3. Scope

### 3.1 In Scope

* Consumption of findings from:
  * Network Discovery Domain
  * Identity Discovery Domain
  * Application / Service Discovery Domain
  * Telemetry / Logging Exposure Discovery Domain
* Deterministic, rule-based inference of host configuration posture
* Generation of normalized findings using the DiscoveryFinding schema
* Identification of security-relevant configuration indicators based on previously observed discovery signals
* Consumption of explicitly authorized governed upstream evidence fields through read-only, append-only, contract-bound handling only
* Exact-match, specification-defined use of producer-owned governed evidence fields where expressly authorized by this specification

### 3.2 Out of Scope

* Any form of direct network interaction
* Port probing or connectivity testing
* Protocol negotiation or enumeration
* Authentication or credential usage
* Modification of upstream findings
* Cross-domain logic coupling outside explicitly authorized consuming behavior
* Normalization, aliasing, synonym handling, heuristic reinterpretation, or semantic expansion of governed upstream evidence fields
* Consumption of future-domain outputs unless and until explicitly authorized by a future controlled specification update
* Any modification to previously validated upstream or foundational components without formal change control

## 4. Technical Approach

The Host Configuration Discovery Domain extends the Core Framework module contract as an inference-only domain. This domain relies on orchestrator-provided prior findings, enabled by the controlled Core Framework contract for authorized downstream consumption of previously produced governed findings. No additional orchestration changes are required beyond this defined mechanism, and no changes are introduced to producing discovery domains by this specification update.

This domain consumes aggregated findings provided by the Discovery Orchestrator and applies deterministic, rule-based evaluation to derive host configuration insights. No network operations, socket interactions, protocol-level activity, authentication behavior, or discovery activity are performed.

This domain consumes only explicitly authorized upstream findings and governed evidence fields defined elsewhere in this specification.

Inference logic evaluates the presence and combination of previously observed authorized discovery signals to identify configuration patterns and bounded host security posture indicators. All outputs are newly generated findings and do not modify, overwrite, suppress, reorder, or reinterpret upstream data in place.

All processing must remain deterministic. Identical input findings must produce identical output findings. No randomness, concurrency, time-dependent behavior, dynamic rule creation, runtime prioritization, or hidden dependency on undocumented upstream behavior is permitted.

The domain receives prior discovery findings from the Discovery Orchestrator as a read-only, append-only collection.

- All inference logic operates exclusively on provided findings
- Only explicitly authorized upstream categories and governed evidence fields may be consumed by this domain
- Presence of additional upstream findings does not, by itself, authorize their use by this domain

## 11. Inference Strategy

The Host Configuration Discovery Domain applies a deterministic, rule-based methodology to derive host posture insights from previously observed discovery signals.

### 11.1 Inference Design Principles

All inference logic must adhere to the following constraints:

* **Input-only processing:** Only consumes findings provided by the orchestrator
* **No network interaction:** No sockets, probes, or connectivity checks
* **Deterministic evaluation:** Fixed rule ordering with no randomness
* **Read-only input:** Upstream findings must not be modified
* **Traceable evidence:** All outputs must reference source findings
* **No semantic expansion beyond approved rules:** This domain must not derive broader host meaning than the exact rule-defined posture meaning authorized by this specification

Inference execution SHALL follow a fixed, rule-based evaluation model:

* Rules are evaluated sequentially in a predefined, deterministic order
* Each rule independently evaluates the full set of prior findings
* Rules MUST NOT modify shared state or intermediate results
* No implicit or dynamic inter-rule dependencies are permitted

The inference engine SHALL:

* Consume all prior findings as a read-only dataset
* Apply all rules consistently for a given execution target
* Emit only normalized `DiscoveryFinding` objects
* Produce identical outputs when provided identical inputs

No dynamic rule creation, runtime prioritization, randomized processing, or conditional reordering of rule execution is permitted.

All inference behavior implemented in code MUST be explicitly defined within this specification prior to implementation.

### 11.2 Reachability Signal Semantics (Normalization)

This section defines the authoritative semantics for reachability-related signals used by the Host Configuration Discovery Domain. These definitions are required to eliminate ambiguity between ICMP-based host presence and TCP-based service exposure and MUST be applied consistently across all inference rules.

---

#### 11.2.1 ICMP-Based Host Presence

- **Source Domain:** network  
- **Category:** host_presence  

**Definition:**
Represents the result of an ICMP echo request (ping) from the discovery vantage point.

**Constraints:**
- MUST reflect ICMP behavior only  
- MUST NOT be interpreted as general host reachability  
- MUST NOT imply TCP connectivity or service availability  
- Absence of ICMP response MUST NOT be interpreted as host inaccessibility  

---

#### 11.2.2 TCP-Based Service Exposure

- **Source Domains:**
  - network → open_port  
  - identity → identity_service_exposed  
  - application_service → application_service_exposed
  - telemetry_logging → telemetry_logging_exposed

**Definition:**
Represents successful TCP connect-only interaction on a specific port.

**Constraints:**
- MUST be port-specific only  
- MUST NOT imply general host reachability  
- MUST NOT imply ICMP reachability  
- MUST NOT perform protocol negotiation, authentication, or payload exchange  
- MUST comply with 01.07 Protocol Interaction Standard (connect-only)  

---

#### 11.2.3 Signal Independence

ICMP-based host presence and TCP-based service exposure are independent signals.

The following rules apply:

- ICMP success does not imply TCP accessibility  
- TCP success does not imply ICMP reachability  
- Absence of one signal does not invalidate the other  
- Discovery domains MUST NOT combine or interpret these signals together  

---

#### 11.2.4 TCP-Derived Reachability (Inference Concept)

**Definition:**
A host is considered TCP-reachable when one or more TCP connect-only service exposure findings are present for that target.

**Constraints:**
- This is an inference-only concept and MUST NOT be emitted by discovery domains  
- MUST be derived solely from existing TCP-based findings  
- MUST NOT be generalized beyond observed service exposure  
- MUST be evaluated deterministically using rule-based logic  

---

#### 11.2.5 Inference Boundary Enforcement

Discovery domains (Network, Identity, Application / Service, and Telemetry / Logging Exposure) MUST NOT perform any inference based on combinations of ICMP and TCP signals.

Specifically, discovery modules MUST NOT:

- Combine ICMP and TCP signals  
- Infer host reachability from TCP success  
- Infer service availability from ICMP response  
- Derive host-level conclusions from multi-signal relationships  

All such interpretation is strictly reserved for inference domains (e.g., Host Configuration Domain).

---

#### 11.2.6 Relationship Interpretation Matrix

| ICMP Result | TCP Result | Interpretation |
|------------|-----------|----------------|
| Reachable  | None      | ICMP-reachable only |
| Not reachable | Present | TCP-reachable only |
| Reachable  | Present   | ICMP and TCP signals independently valid |
| Not reachable | None   | No ICMP or TCP reachability signals observed |

These interpretations are informational and MUST NOT alter raw signal meaning.

### 11.4 Evidence Collection Strategy

Each inferred finding must include:

* Source findings contributing to the inference
* Applied inference rule identifier
* Human-readable rationale explaining the conclusion

Evidence must be derived strictly from previously observed findings and must not introduce new external data.

### 11.5 Confidence Assignment

Confidence must be statically assigned based on rule strength and signal reliability.

* High (0.8–0.95): Strong indicators of host configuration state, including highly reliable single-service signals or multiple corroborating signals
* Medium (0.5–0.79): Moderate indicators based on single observable signals or partial context
* Low (<0.5): Weak or indirect signals with limited confidence

Confidence assignment MUST be deterministic and defined per rule.

No dynamic or probabilistic adjustment is permitted.

### 11.6 Processing Controls

To maintain enterprise-safe and deterministic behavior:

* Rules must execute in fixed order
* Duplicate inference generation must be prevented
* Output ordering must remain consistent
* Identical input must always produce identical output
* Only explicitly authorized upstream categories and governed evidence fields may participate in rule evaluation
* Governed evidence handling must remain exact-match, read-only, and specification-bound across all authorized producer domains

#### Rule Execution Ordering

All inference rules SHALL be evaluated in a fixed, predefined order.

* Rules are evaluated sequentially (no parallel execution)
* Each rule evaluates against the full set of prior findings
* Rule execution order MUST remain consistent across all runs
* Output findings MUST be emitted in the same order as rule evaluation
* Rule evaluation MUST NOT depend on undocumented producer behavior, incidental formatting, or cross-field aliasing
* Authorized governed evidence fields MUST be consumed only through the exact rule logic defined in Section 11.7

No dynamic reordering, prioritization, conditional skipping, heuristic remapping, or runtime reinterpretation of governed evidence is permitted unless explicitly defined in this specification.

This ensures:

* Deterministic output generation
* Consistent behavior across environments
* Full auditability of inference logic
* Controlled producer / consumer contract enforcement across all authorized upstream domains

#### 11.7.5 Duplicate Prevention

Each rule may emit only one finding per execution target.

If multiple upstream findings satisfy a rule:

* All may be referenced in evidence
* Only one derived finding is produced

Deduplication applies only within an individual rule. Cross-rule consolidation or suppression of findings is not permitted.
