## 1. Purpose

This Architecture Design Document (ADD) defines the complete system architecture for the Enterprise Security Discovery Orchestration Framework. It establishes the structural design, component boundaries, and interaction model of the system.

This document is a control artifact and defines the full architectural model of the system independent of implementation sequence or development phase.

---

## 2. Architectural Principles

The system architecture is governed by the following principles:

- **Agentless Operation:** No agents are deployed on target systems
- **Non-Exploitative Design:** No exploitation, credential use, or intrusive techniques are permitted
- **Deterministic Execution:** All processing must be predictable and repeatable
- **Domain Isolation:** Each domain operates independently with no direct coupling
- **Separation of Concerns:** Distinct responsibilities across system layers
- **Standardized Output:** All findings conform to a unified schema
- **Producer-Owned Semantics:** Findings retain the domain, category, evidence fields, and semantic ownership assigned by the authorized producing layer or domain
- **Bounded TCP Inventory Control:** TCP-based discovery operates within the approved project-scoped TCP port inventory defined by the TRD and applicable governing specifications
- **Scalability:** Architecture supports expansion across domains and system layers
- **Security by Design:** All components adhere to strict security constraints defined in the TRD

---

## 3. High-Level Architecture

The system is composed of layered components that operate in a structured, deterministic, and responsibility-bound workflow.

1. Core Framework Layer  
2. Discovery Layer  
3. Inference Layer  
4. Correlation Layer  
5. Output Layer  
6. Control Layer  
7. Interface Layer  
8. Deployment Layer  

Each layer is logically isolated and interacts only through defined interfaces and approved contracts.

Layer responsibilities are non-overlapping and SHALL remain explicitly bounded as follows:

- Discovery Layer: direct observation only
- Inference Layer: derived conclusions from previously produced governed findings only
- Correlation Layer: cross-domain relationship analysis on previously produced governed findings only
- Core Framework Layer: orchestration, sequencing, aggregation, and failure isolation only
- Control Layer: controlled input handling and invocation only
- Output Layer: presentation and serialization only

Producer / consumer separation MUST be preserved across all governed findings handling.

No layer may exceed its authorized responsibility boundary or rely on hidden coupling to other layers or domains.

The architecture recognizes the approved project-scoped TCP port inventory as the bounded TCP discovery control model defined by the TRD. This inventory establishes the maximum TCP port scope available to authorized discovery domains unless expanded through governed change control.

Within this architecture, Network Discovery may operate as the generic TCP reachability producer for `network/open_port` findings across the approved project-scoped TCP port inventory. Specialized discovery domains may operate as bounded direct-observation producers for domain-specific findings on approved subsets of the same inventory.

Overlapping generic and specialized findings on the same TCP port are architecturally permitted only when each finding preserves distinct producer ownership, domain, category, evidence fields, and semantic meaning. Such overlap does not authorize inference, correlation, protocol validation, response interpretation, suppression, merging, or semantic reconciliation by discovery domains, the Core Framework, or the Output Layer.

## 4. Core Framework Layer

The Core Framework Layer provides the foundational structure for module orchestration, sequencing, aggregation, shared contracts, and failure isolation across the system.

### Responsibilities:

- Orchestration of module execution
- Deterministic execution control and sequencing
- Management of registered modules
- Aggregation of findings and events
- Propagation of authorized inputs through approved contracts only
- Enforcement of standardized data structures
- Preservation of structured failure isolation

### Constraints:

- No discovery logic
- No inference logic
- No correlation logic
- No normalization, reinterpretation, suppression, or mutation of domain findings
- No suppression, merging, deduplication, or semantic reconciliation of overlapping producer-owned findings
- No domain-specific business rules

### Core Components:

- Discovery Orchestrator
- Discovery Context
- Discovery Module Interface
- Discovery Finding Schema

#### Discovery Module Interface

The Discovery Module Interface defines the standardized contract used by authorized system modules operating within approved architectural responsibility boundaries.

Each module:

- Receives the shared execution context provided by the Core Framework
- Produces a list of normalized `DiscoveryFinding` objects

Where explicitly authorized by architecture and consuming-layer specification, a module may also receive aggregated findings produced by previously executed modules in the orchestration sequence.

Aggregated findings provided to an authorized consuming module:

- Represent an append-only collection of prior governed results
- Are treated as **read-only and immutable**
- Are **contract-bound** and authoritative within their governed meaning
- Must not be modified, overwritten, reordered, filtered in place, normalized, aliased, or reinterpreted except where explicitly authorized by an approved consuming specification
- Must not be used by discovery domains to perform inference, correlation, suppression, or higher-order meaning assignment

Modules must operate independently and must not rely on external state outside of the provided context and approved inputs.

#### Execution Model

- Modules are executed sequentially by the Discovery Orchestrator
- Each module produces findings that are aggregated and made available to subsequent processing only through approved contracts
- The Discovery Orchestrator passes findings without reinterpretation
- Overlapping findings produced by authorized domains are aggregated as distinct governed findings and are not merged, suppressed, or reinterpreted by the Discovery Orchestrator
- Findings may serve as output artifacts for reporting and analysis and, where explicitly authorized, as input signals for inference or correlation layers

This model preserves deterministic execution while maintaining strict separation between discovery, inference, correlation, control, and reporting responsibilities.

## 5. Discovery Layer

The Discovery Layer is responsible only for **direct observation** of externally observable system characteristics obtained through approved, agentless, and non-exploitative interaction models.

### Domains Included:

- Network Discovery Domain
- Identity Discovery Domain
- Application / Service Discovery Domain
- Telemetry / Logging Exposure Discovery Domain

### Responsibilities:

- Perform only domain-approved observation activities
- Produce governed findings using the normalized `DiscoveryFinding` schema
- Emit direct observations only
- Preserve deterministic execution and evidence structure
- Remain independent of downstream interpretation, inference, and correlation logic

### Constraints:

- No authentication or credential usage
- No protocol negotiation or enumeration beyond defined constraints
- No direct system modification
- No inference reserved for inference layers
- No cross-domain correlation
- No consumption of upstream findings to perform higher-order meaning assignment
- No mutation, normalization, relabeling, suppression, or extension of upstream evidence
- Strict adherence to domain isolation

Each discovery domain operates as an isolated producer of governed findings through its own bounded observation model and MUST remain independent from consuming-layer interpretation.

Future discovery domains, including Application / Service Discovery and Telemetry / Logging Exposure Discovery, MUST remain discovery-only unless a separately approved inference or correlation capability is defined.

### Bounded TCP Inventory and Discovery Overlap Model

The Discovery Layer SHALL operate within the approved project-scoped TCP port inventory defined by the TRD and applicable governing specifications.

The approved project-scoped TCP port inventory is an architecture-recognized control concept that bounds TCP connect-only discovery across the framework. It does not, by itself, assign domain-specific semantic meaning to a port and does not authorize behavior outside the approved interaction model.

Network Discovery is the authorized generic TCP reachability producer for `network/open_port` findings where TCP probing is permitted across the approved project-scoped TCP port inventory. A `network/open_port` finding represents only bounded generic TCP exposure observed through connect-only interaction.

Specialized discovery domains, including Identity Discovery, Application / Service Discovery, and Telemetry / Logging Exposure Discovery, may produce domain-specific direct-observation findings on approved bounded subsets of the same project-scoped TCP port inventory when authorized by their governing specifications.

A specialized discovery finding represents only the producing domain’s bounded direct-observation classification. It MUST retain its own approved domain, category, evidence fields, controlled vocabulary, and semantic ownership.

Overlapping findings on the same TCP port are permitted only when:

- each finding is produced independently by its authorized producing domain
- each producing domain uses only its approved bounded observation model
- each finding preserves distinct domain, category, evidence fields, and semantic ownership
- no producing domain suppresses, mutates, relabels, normalizes, or reinterprets another domain’s findings
- no overlap is used to authorize inference, correlation, protocol validation, payload transmission, response parsing, TLS handshakes, banner grabbing, authentication, enumeration, or adaptive probing

Discovery-domain overlap MUST NOT imply protocol correctness, service health, authentication state, configuration posture, successful application-layer interaction, or generalized host reachability.

## 6. Inference Layer

The Inference Layer is responsible for **derived conclusions** based solely on previously produced governed findings.

### Capabilities Included:

- Host Configuration Discovery Domain

### Responsibilities:

- Consume upstream findings as read-only, append-only, contract-bound input
- Apply deterministic, specification-defined logic only
- Derive higher-level conclusions from previously observed signals
- Produce new derived findings without modifying upstream findings
- Preserve traceability from derived outputs to source findings

### Constraints:

- No direct network interaction
- No probing, socket usage, or protocol interaction
- No dependency on external systems or external data sources
- No modification, overwrite, reordering, or in-place reinterpretation of source findings
- No hidden dependencies on undocumented upstream behavior

This layer enhances visibility through deterministic inference without expanding the attack surface or altering governed upstream outputs.

---

## 7. Correlation Layer

The Correlation Layer is responsible for **cross-domain relationship analysis** across previously produced governed findings.

### Responsibilities:

- Cross-domain correlation of findings
- Identification of relationships and patterns across domains
- Aggregation of signals into higher-confidence composite insights
- Preservation of traceability between correlated outputs and source findings

### Constraints:

- Operates solely on existing governed findings
- Does not perform discovery activity or probing
- Does not perform direct network interaction
- Does not modify, overwrite, or reclassify source findings in place
- Maintains deterministic processing

This layer remains distinct from inference by focusing on cross-domain relationship analysis rather than domain-local derived posture conclusions.

---

## 8. Output Layer

The Output Layer is responsible only for **presentation and serialization** of governed findings and events into approved, consumable formats.

### Responsibilities:

- Deterministic serialization of findings and events
- Report generation in approved output formats
- Presentation of governed outputs without altering their meaning
- Preservation of evidence integrity and traceability

### Constraints:

- No discovery logic
- No inference logic
- No correlation logic
- No alteration of the semantic meaning of findings
- No mutation of evidence, category values, or governed vocabulary during presentation
- No reconciliation of overlapping generic and specialized findings into new semantic conclusions
- No suppression of governed findings because another domain produced a related finding
- No treatment of report sections, headings, tables, or formatting behavior as the technical source of truth
- No redefinition of exposure semantics based on presentation needs

### Output Formats:

- Structured JSON output
- Human-readable reports (e.g., Markdown)

All outputs must reflect the standardized finding schema and preserve governed output meaning exactly as produced by authorized upstream layers.

The structured `DiscoveryFinding` output remains the authoritative technical source of truth for discovery evidence. Human-readable reports may summarize, group, or format findings for readability, but such presentation behavior MUST remain traceable to the underlying governed findings and MUST NOT replace producer-owned finding semantics.

Where generic `network/open_port` findings and specialized discovery findings exist for the same TCP port, the Output Layer may present them in a human-readable manner, but it MUST NOT merge them into a single semantic conclusion, suppress either finding, or determine exposure completeness based on report section membership.

---

## 9. Control Layer

The Control Layer is responsible only for **controlled execution input handling and invocation**.

### Responsibilities:

- Accept and validate approved operator inputs
- Construct authorized execution context
- Invoke the orchestration framework
- Remain free of domain-specific business logic

### Constraints:

- No discovery logic
- No inference logic
- No correlation logic
- No selection or reinterpretation of findings based on domain meaning
- No execution branching tied to domain conclusions unless formally approved by design
- No coupling to domain-specific behavior

This layer provides controlled system entry and execution management without altering architectural responsibility boundaries.

---

## 10. Interface Layer

The Interface Layer enables interaction between the system and external consumers.

### Responsibilities:

- Integration with external tools and systems
- Exposure of system outputs
- Support for extensibility and interoperability

### Characteristics:

- Read-only interaction with system outputs
- No direct control over internal system behavior
- Preservation of governed output meaning without reinterpretation, normalization, suppression, or semantic enrichment
- No discovery, inference, correlation, orchestration, control, or reporting business logic

---

## 11. Deployment Layer

The Deployment Layer defines how the system is packaged, secured, and operated.

### Responsibilities:

- Packaging and distribution
- Environment configuration
- Security hardening
- Operational readiness

### Characteristics:

- Supports controlled and secure execution environments
- Ensures compliance with enterprise security standards

---

## 12. Data Flow Overview

The system follows a structured, responsibility-bound data flow:

1. Input is received through the Control Layer  
2. Execution is coordinated by the Core Framework  
3. Discovery domains produce governed direct-observation findings, including distinct generic and specialized TCP findings where overlap is authorized  
4. Authorized inference capabilities consume existing findings through approved read-only, append-only, contract-bound inputs  
5. Authorized correlation capabilities perform cross-domain analysis on existing governed findings  
6. Results are serialized and presented by the Output Layer without semantic alteration  
7. Outputs are exposed via the Interface Layer  

All data flows through standardized interfaces, preserve governed finding meaning, and adhere to deterministic processing rules.

## 13. Architectural Constraints

The architecture must comply with the following constraints:

- Agentless operation only  
- Non-exploitative techniques only  
- No credential usage  
- No unauthorized system interaction  
- Deterministic execution required  
- Strict domain isolation enforced  
- Discovery domains must remain discovery-only and emit direct observations only  
- Inference capabilities must remain inference-only and operate only on previously produced governed findings  
- Correlation capabilities must remain correlation-only and operate only on previously produced governed findings  
- The Core Framework must remain orchestration-only  
- The Control Layer must remain control-only  
- The Output Layer must remain presentation-only  
- The Interface Layer must remain interface-only and must not reinterpret governed outputs  
- The Deployment Layer must remain deployment-oriented and must not introduce discovery, inference, correlation, orchestration, control, reporting, or interface business logic  
- Output / Reporting must not treat report sections, headings, tables, or formatting behavior as the technical source of truth for exposure semantics  
- Producer / consumer separation must be preserved across all governed findings handling  
- Upstream findings consumed by authorized downstream layers must be treated as read-only, append-only, and contract-bound  
- Hidden coupling between layers or domains is prohibited  
- TCP-based discovery must remain bounded by the approved project-scoped TCP port inventory defined by the TRD and applicable governing specifications  
- Network Discovery may produce generic `network/open_port` findings across the approved project-scoped TCP port inventory only within approved connect-only interaction constraints  
- Specialized discovery domains may produce domain-specific direct-observation findings on approved bounded subsets of the same inventory  
- Overlapping generic and specialized TCP findings must preserve distinct producer ownership, domain, category, evidence fields, and semantic meaning  
- Overlap must not be used to authorize inference, correlation, protocol validation, response interpretation, suppression, merging, or semantic reconciliation inside discovery domains, the Core Framework, or the Output Layer  

These constraints are defined by the Technical Requirements Document (TRD) and reinforced by governing architectural control standards. They are mandatory for all components.

## 14. Alignment with Project Scope

This architecture supports the full system scope defined in the Project Initiation Document (PID) and Project Management Plan (PMP), covering all capability layers and system components within approved project boundaries.

This includes current and future discovery capabilities, inference capabilities, correlation capabilities, control interfaces, reporting components, external interface layers, and deployment-readiness concerns, provided each remains within its approved architectural responsibility boundary.

Future capability expansion, including Application / Service Discovery and Telemetry / Logging Exposure Discovery, MUST remain aligned with this architectural model and MUST NOT require exceptions to domain-isolation or layer-responsibility controls unless separately approved through formal change control.

This document defines the complete architectural model of the system and is not intended to be modified based on implementation progress or milestone development. Architectural updates must be driven by approved control changes, scope changes, or formally governed design corrections.
