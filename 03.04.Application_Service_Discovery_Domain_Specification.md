## 1. Purpose

The Application / Service Discovery Domain defines a deterministic, discovery-only domain responsible for producing direct-observation findings for externally reachable application-adjacent service interfaces observed through approved connect-only TCP interaction on a bounded approved port set.

This domain operates in an agentless, non-exploitative manner and produces normalized `DiscoveryFinding` outputs aligned with the Core Framework schema.

This domain exists to model bounded application/service interface exposure without introducing protocol interaction, banner collection, handshake inspection, metadata extraction, or higher-order interpretation.

---

## 2. Architectural Boundary Statement

This domain is a discovery-only producer of direct-observation findings derived solely from connect-only TCP exposure on approved application/service ports, with no negotiation, no banner grabbing, no handshake inspection, no metadata extraction, and no inference.

This boundary is authoritative for all implementation, validation, and downstream consumption related to this domain.

---

## 3. Vantage Point Assumptions

* No valid credentials are supplied.
* Discovery is executed from an external or minimally privileged network position.
* No exploitation, brute force, authentication bypass, or active interrogation techniques are used.
* Observations are limited to externally observable TCP connectivity signals on approved application/service ports.
* No target response content is read, parsed, or interpreted.
* No upstream findings are consumed by this domain.

---

## 4. Scope

### 4.1 In Scope

* Detection of externally reachable application/service interfaces via successful connect-only TCP establishment on approved application/service ports.
* Deterministic classification of approved application/service interface exposure based on static port-to-classification association.
* Production of normalized `DiscoveryFinding` outputs using approved categories and governed evidence fields.
* Low-noise, fixed-order probing of the approved M7 application/service port set.

### 4.2 Out of Scope

* Generic network exposure detection outside the approved application/service bounded port set.
* Identity-related service exposure detection.
* Telemetry / logging endpoint discovery.
* Authentication attempts of any kind.
* HTTP requests, headers, or method-based interaction.
* TLS handshake initiation or inspection.
* Banner grabbing.
* Response reading or parsing.
* Metadata extraction.
* Product, vendor, framework, or version fingerprinting.
* Inference, correlation, or posture interpretation.
* Consumption of upstream findings for meaning assignment.
* Any modification to frozen validated components without formal change control.

---

## 5. Relationship to Existing Domains

### 5.1 Network Discovery Domain

The Network Discovery Domain remains the authoritative producer of:

* `host_presence`
* `open_port`
* Generic bounded network exposure signals
* Its own governed `service_hint` handling within its producing contract

This Application / Service Discovery Domain does not replace, rename, or reinterpret Network Discovery behavior.

This domain MUST NOT emit `open_port` and MUST NOT duplicate generic network exposure semantics under alternate naming.

### 5.2 Identity Discovery Domain

Identity Discovery remains the authoritative producer of identity-related service exposure on approved identity ports and the authoritative owner of its governed `service_hint` contract.

This domain MUST NOT reuse Identity Discovery governed vocabulary, identity categories, or identity semantic ownership.

### 5.3 Host Configuration Discovery Domain

Host Configuration remains an inference-only downstream consumer domain.

Application / Service Discovery is a discovery producer only and MUST NOT perform inference that belongs to Host Configuration or any future correlation capability.

---

## 6. Technical Approach

The Application / Service Discovery Domain extends the existing Core Framework module contract as an isolated discovery producer.

The domain performs deterministic TCP connect-only checks against an approved bounded application/service port list. Findings are produced only when a TCP connection is successfully established. No application-layer data is transmitted, no socket data is read, no protocol negotiation occurs, and no response-derived interpretation is performed.

All classification in this domain is static and deterministic. Classification is derived solely from approved destination-port association and successful TCP connection establishment.

As a producing discovery domain, this domain MUST remain discovery-only. It MUST NOT:

* consume upstream findings
* infer host posture
* correlate findings across domains
* encode hidden inference in evidence fields
* depend on downstream interpretation to define its output meaning

All interactions MUST use standard operating system socket mechanisms, bounded static timeouts, fixed execution order, and structured exception handling consistent with approved project constraints.

---

## 7. Runtime Position and Execution Ordering

This domain is a discovery producer and is expected to execute after existing discovery producers that precede it in the approved milestone sequence and before any authorized inference consumer that follows discovery producers.

For initial M7 architectural planning, the expected deterministic runtime ordering is:

1. Network Discovery Domain
2. Identity Discovery Domain
3. Application / Service Discovery Domain
4. Host Configuration Discovery Domain

This ordering assumption exists to preserve architectural clarity and does not authorize this domain to consume prior findings.

This domain MUST use only local execution context and approved local configuration.

---

## 8. Approved Initial Port Scope

The initial approved Application / Service Discovery M7 scope is limited to the following TCP ports:

* TCP/80 — `HTTP`
* TCP/443 — `HTTPS`
* TCP/8080 — `HTTP-Alt`
* TCP/8443 — `HTTPS-Alt`

No additional ports are authorized for this domain unless this specification is updated through controlled change.

### 8.1 Rationale for Inclusion

These ports are approved as application-adjacent service interface ports suitable for bounded connect-only exposure observation without identity-domain overlap.

### 8.2 Rationale for Exclusion

The following are excluded from initial M7 Application / Service Discovery scope:

* Identity-related ports already governed by Identity Discovery
* Generic network ports not clearly application-interface focused
* Telemetry / logging endpoints reserved for future domain-specific treatment
* Any port requiring broader semantic interpretation not yet formally governed

---

## 9. Probe Design Principles

All application/service probes MUST adhere to the following constraints:

* **Connect-only interaction:** Probes are limited strictly to TCP connection establishment.
* **No application-layer interaction:** No HTTP requests, TLS handshakes, banners, payloads, or protocol exchanges are permitted.
* **No credential transmission:** The module must never send usernames, passwords, tokens, cookies, headers, certificates, or authentication material.
* **Low-noise operation:** Timeouts and retry behavior must minimize network noise.
* **Deterministic behavior:** Identical inputs and environmental conditions must yield identical outputs.
* **Failure isolation:** Probe failures must not interrupt orchestrator execution.

---

## 10. Governed Classification Field

### 10.1 Field Name

`application_service_hint`

### 10.2 Definition

`application_service_hint` represents a deterministic, producer-owned classification of the approved application/service interface associated with a successfully established TCP connection on an Application / Service Discovery approved port.

It is derived strictly from:

* The destination port
* The fact that a TCP connection was successfully established
* The approved static port-to-classification mapping defined by this specification

No protocol interaction, negotiation, validation, response parsing, or metadata inspection is performed.

### 10.3 Constraints

`application_service_hint` MUST:

* Be derived solely from connect-only TCP success signals on this domain’s approved port set
* Be deterministic and consistent across executions
* Use a controlled, predefined vocabulary
* Represent classification only

`application_service_hint` MUST NOT:

* Reuse or alias `service_hint`
* Encode reachability semantics beyond the producing contract
* Imply protocol correctness, successful application negotiation, or service health
* Imply authentication state, TLS state, certificate validity, or response behavior
* Include inferred host roles or configurations
* Be dynamically generated or environment-dependent
* Be derived from response content, banners, handshakes, or metadata

### 10.4 Allowed Values (Application / Service Domain)

The following values are the authoritative controlled vocabulary for initial M7 Application / Service Discovery:

* `"HTTP"`
* `"HTTPS"`
* `"HTTP-Alt"`
* `"HTTPS-Alt"`

No additional values are permitted without controlled specification update.

### 10.5 Cross-Domain Contract Requirements

This domain is the authoritative producing owner of `application_service_hint`.

Producing logic MUST:

* Emit only approved values
* Treat `application_service_hint` as a governed evidence field
* Avoid hidden inference, role claims, posture claims, product identification, or downstream-layer meaning injection

Consuming domains MUST treat `application_service_hint` as:

* Read-only
* Authoritative
* Exact-match only
* Non-derivable outside approved consuming specifications

No domain may normalize, alias, reinterpret, extend, or silently remap the meaning of `application_service_hint`.

### 10.6 Determinism Requirements

For identical inputs:

* `application_service_hint` MUST always resolve to the same value
* Mapping between approved port and `application_service_hint` MUST remain fixed
* No conditional or environment-based variation is allowed

## 11. Finding Model

### 11.1 Output Domain

All findings generated by this domain MUST use:

* `domain = "application_service"`

### 11.2 Approved Output Category

The following `DiscoveryFinding.category` value is authorized for initial M7 implementation:

* `application_service_exposed`

No alternate category names, synonyms, or additional categories are authorized without controlled update.

### 11.3 Category Determination Constraints

* Findings are produced only upon successful connect-only TCP establishment on an approved application/service port.
* Category assignment MUST be deterministic.
* Category values MUST NOT be response-derived or dynamically generated.
* This domain MUST NOT emit generic `open_port`.

---

## 12. Evidence Collection Strategy

A probe may generate a finding when the following condition is observed:

* Successful TCP connection indicating application/service interface exposure on an approved application/service port.

Evidence must remain strictly observational and derived from TCP connectivity only.

Permitted evidence elements for positive findings include:

* `method`
* `port`
* `application_service_hint`
* `timeout_s`
* Optional controlled explanatory note consistent with this specification

Evidence MUST NOT include:

* response content
* banners
* headers
* handshake artifacts
* metadata
* certificate details
* vendor or product identification
* inferred posture or role semantics

---

## 13. Timeout and Safety Controls

To maintain enterprise-safe behavior:

* Default socket timeout must remain low (≤ 1–2 seconds).
* No aggressive retry loops are permitted.
* Parallel probing must be avoided unless explicitly authorized in a future controlled update.
* All socket operations must be wrapped in structured exception handling.
* No payload transmission is permitted after connection establishment.
* Connection must terminate immediately once success or failure is determined.

---

## 14. Responsibility Boundary Enforcement

### 14.1 What Network Discovery Owns

Network Discovery owns:

* host presence observation
* generic TCP exposure observation
* `open_port`
* generic network-domain port/service classification within its own producing contract

### 14.2 What Application / Service Discovery Owns

Application / Service Discovery owns:

* direct-observation findings for successful connect-only exposure on its approved bounded application/service port set
* `application_service_exposed`
* `application_service_hint` as its governed producer-owned classification field

### 14.3 Overlap Prevention Rules

To prevent semantic duplication:

* This domain MUST NOT emit `open_port`
* This domain MUST NOT reuse `service_hint`
* This domain MUST NOT classify identity-related ports
* This domain MUST remain limited to its approved bounded scope
* This domain MUST NOT reinterpret Network or Identity outputs
* This domain MUST NOT depend on Network or Identity findings to suppress or produce its own outputs

## 18. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Semantic duplication with Network Discovery | High | Enforce separate category, bounded port set, and producer-owned governed field |
| Hidden coupling through reused vocabulary | High | Prohibit reuse of `service_hint`; define `application_service_hint` as separate governed contract |
| Unauthorized protocol interaction | High | Enforce 01.07 connect-only methodology and hard prohibitions |
| Scope creep into inference or product fingerprinting | High | Preserve discovery-only boundary and forbid metadata extraction |
| Non-deterministic behavior | Medium | Enforce fixed probe list, fixed order, static mapping, fixed timeout |
| Architectural drift into telemetry or identity scope | Medium | Preserve bounded initial port list and explicit exclusion rules |
