## 1. Purpose

The Correlation Domain defines the governed Milestone M8 correlation capability for the Enterprise Security Discovery Orchestration Framework.

This domain performs deterministic, traceable cross-domain relationship analysis across previously produced governed findings. It operates only on existing findings produced by authorized discovery and inference capabilities and produces correlation-owned outputs without modifying, suppressing, merging, deduplicating, normalizing, reclassifying, or reinterpreting upstream findings in place.

This specification governs the full M8 Correlation Domain capability. Implementation may proceed in controlled phases as defined in this document. Phased implementation does not change the architectural identity, responsibility boundary, or final intended capability of this domain.

---

## 2. Architectural Boundary Statement

The Correlation Domain is a correlation-only consuming capability.

The domain consumes existing governed findings as read-only, append-only, contract-bound input and applies deterministic, specification-defined correlation logic to identify approved relationships between independent semantic domains.

The Correlation Domain MUST NOT:

* Perform discovery activity
* Perform direct network interaction
* Open sockets
* Probe ports
* Authenticate
* Negotiate protocols
* Parse responses
* Perform banner grabbing
* Perform TLS handshakes
* Modify upstream findings
* Suppress upstream findings
* Reclassify source findings in place
* Duplicate Host Configuration inference responsibility
* Treat expected Network/specialized discovery overlap as correlation by itself

This domain is distinct from discovery domains, Host Configuration inference, Core Framework orchestration, CLI / Control, and Output / Reporting.

---

## 3. Scope

### 3.1 In Scope

The following activities are in scope:

* Consumption of previously produced governed findings through orchestrator-provided inputs
* Deterministic correlation across authorized governed findings
* Identification of approved relationships between independent semantic domains
* Generation of correlation-owned outputs using normalized `DiscoveryFinding` records unless a later output structure is explicitly defined in this specification
* Preservation of source traceability for every correlation output
* Fixed, governed rule ordering
* Phased implementation from initial specialized-domain correlation through full M8 correlation capability

### 3.2 Out of Scope

The following activities are out of scope:

* Discovery, probing, socket usage, protocol interaction, authentication, enumeration, response parsing, TLS handshake inspection, or target state modification
* Risk scoring, severity scoring, prioritization, exploitability assessment, or attack-path generation
* External enrichment or use of non-framework data sources
* Mutation, suppression, deduplication, merging, normalization, reclassification, or semantic reconciliation of upstream findings
* Business logic leakage into Core Framework, discovery domains, Host Configuration, CLI / Control, Output / Reporting, Interface Layer, or Deployment Layer

---

## 4. Relationship to Other Domains and Layers

### 4.1 Network Discovery Domain

Network Discovery remains the authoritative producer of `network/host_presence`, `network/open_port`, and Network-owned `service_hint` values.

The Correlation Domain MUST NOT treat expected overlap between Network `open_port` findings and specialized discovery-domain findings as a correlation insight by itself.

Network findings are not used by the initial specialized-domain implementation phase. Network findings are part of the full M8 capability where rule-defined Network use is limited to non-default relationships such as unclassified Network exposure coexistence.

### 4.2 Identity Discovery Domain

Identity Discovery remains the authoritative producer of `identity/identity_service_exposed` findings and `identity_service_hint` values.

The Correlation Domain may consume Identity findings only through approved correlation rules and MUST preserve Identity-owned semantics.

### 4.3 Application / Service Discovery Domain

Application / Service Discovery remains the authoritative producer of `application_service/application_service_exposed` findings and `application_service_hint` values.

The Correlation Domain may consume Application / Service findings only through approved correlation rules and MUST preserve Application / Service-owned semantics.

### 4.4 Telemetry / Logging Exposure Discovery Domain

Telemetry / Logging Exposure Discovery remains the authoritative producer of `telemetry_logging/telemetry_logging_exposed` findings and `telemetry_logging_hint` values.

The Correlation Domain may consume Telemetry / Logging findings only through approved correlation rules and MUST preserve Telemetry / Logging-owned semantics.

### 4.5 Host Configuration Discovery Domain

Host Configuration remains the authorized inference-only consumer domain for host-level posture inference.

The Correlation Domain may consume Host Configuration findings as bounded posture context only where a rule avoids circular correlation. A Host Configuration finding MUST NOT be treated as an independent correlation signal when the proposed correlation merely relates that Host finding back to one of its own source domains.

The Correlation Domain MUST NOT recreate Host Configuration inference rules, duplicate Host Configuration output categories, or emit correlation findings for relationships already represented by approved Host Configuration inference outputs.

### 4.6 Core, CLI, and Output Layers

Core Framework remains orchestration-only. CLI / Control remains input and invocation-only. Output / Reporting remains presentation-only.

Correlation logic belongs only in the Correlation Domain.

Output / Reporting may serialize or present correlation findings but MUST NOT define, expand, suppress, merge, deduplicate, prioritize, score, normalize, or reinterpret correlation semantics.

---

## 5. Technical Approach

The Correlation Domain extends the existing Core Framework module contract as a downstream consuming module.

The domain receives prior findings from the Discovery Orchestrator and evaluates deterministic rule logic against authorized upstream findings. Correlation processing uses only local in-memory processing of orchestrator-provided findings.

All correlation outputs are append-only. Source findings remain unchanged and authoritative within their producing or deriving contract.

Identical input findings MUST produce identical correlation outputs in identical ordering, except for UTC timestamps generated for new correlation findings.

---

## 6. Runtime Position and Execution Ordering

The approved M8 runtime ordering is:

1. Network Discovery Domain
2. Identity Discovery Domain
3. Application / Service Discovery Domain
4. Telemetry / Logging Exposure Discovery Domain
5. Host Configuration Discovery Domain
6. Correlation Domain

This ordering allows the Correlation Domain to receive the full governed finding set produced by prior modules while preserving strict layer responsibility boundaries.

---

## 7. Input Model

The Correlation Domain receives prior findings from the Discovery Orchestrator as a read-only, append-only collection.

Only approved rule logic may consume upstream findings. The presence of a finding in the prior-finding set does not authorize general-purpose correlation.

### 7.1 Authorized Input Categories

The Correlation Domain is authorized to consume the following categories when referenced by approved rule logic:

* `domain = "identity"`, `category = "identity_service_exposed"`
* `domain = "application_service"`, `category = "application_service_exposed"`
* `domain = "telemetry_logging"`, `category = "telemetry_logging_exposed"`
* `domain = "network"`, `category = "open_port"`, only for Network-inclusive rules defined in this specification
* `domain = "host_configuration"`, approved categories only as bounded context where circular correlation is prevented

### 7.2 Authorized Evidence Fields

Approved correlation rules may consume the following governed evidence fields when needed:

* `evidence.identity_service_hint`
* `evidence.application_service_hint`
* `evidence.telemetry_logging_hint`
* `evidence.service_hint`
* `evidence.port`
* `evidence.method`
* `evidence.timeout_s`
* Host Configuration source-traceability evidence where present and needed to prevent circular correlation

All governed evidence fields MUST be treated as read-only, exact-match, specification-bound values.

---

## 8. Output Model

### 8.1 Initial and Default Output Model

The Correlation Domain emits normalized `DiscoveryFinding` records.

This is the default M8 output model for correlation findings and supports deterministic serialization, reporting, and traceability through the existing framework contract.

### 8.2 Output Domain

All Correlation Domain findings MUST use:

* `domain = "correlation"`

### 8.3 Approved Output Categories

The following categories are approved for M8 correlation outputs:

* `identity_application_telemetry_correlation`
* `identity_application_correlation`
* `identity_telemetry_correlation`
* `application_telemetry_correlation`
* `unclassified_network_identity_correlation`
* `unclassified_network_application_correlation`
* `unclassified_network_telemetry_correlation`
* `unclassified_network_multi_domain_correlation`

Initial implementation may implement only the specialized-domain categories. Network-inclusive categories are part of the planned full M8 capability and may be implemented in a later phase under this specification when the corresponding rules are implemented and validated.

No category may duplicate a Host Configuration inference category or replace, rename, suppress, or reclassify an upstream category.

---

## 9. Evidence Collection Strategy

Each Correlation Domain finding MUST include sufficient evidence to audit the relationship without re-executing the discovery pipeline.

Required evidence elements:

* `correlation_rule_id`
* `correlated_domains`
* `source_findings`
* `rationale`

Optional controlled evidence elements:

* `correlation_type`
* `correlation_scope`
* `notes`

The `source_findings` element MUST preserve traceability to every source finding used by the rule. Initial implementation may preserve source traceability using deterministic source-finding snapshots containing:

* `domain`
* `category`
* `target`
* `evidence`
* `observed_at`

Source-finding snapshots MUST NOT alter the meaning of upstream findings and MUST NOT replace upstream findings as the authoritative source of truth.

---

## 10. Correlation Design Principles

All correlation logic MUST follow these principles:

* Correlation identifies approved relationships among existing governed findings only.
* Correlation findings are append-only.
* Upstream findings are never modified in place.
* Rules execute in fixed order.
* Every finding preserves source traceability.
* Producer-owned semantics remain owned by the producing or deriving domain.
* Host Configuration findings may provide bounded context but MUST NOT create source-recursive correlation.
* Network `open_port` overlap with specialized findings MUST NOT be treated as correlation by itself.
* Correlation MUST NOT duplicate Host Configuration inference categories or rules.

---

## 11. Implementation Phases

### Phase A – Module Scaffolding

* Create `Correlation_Domain.py`.
* Implement the required module interface.
* Register the module after Host Configuration in deterministic runtime order.
* Validate no direct network interaction occurs.
* Validate read-only upstream finding handling.

### Phase B – Specialized-Domain Correlation

* Implement specialized-domain correlation rules from Section 12.1.
* Emit normalized `DiscoveryFinding` records using approved correlation categories.
* Validate tri-domain precedence and pairwise fallback behavior.
* Confirm Network and Host Configuration findings are not used by Phase B rules.

### Phase C – Network-Inclusive Correlation

* Implement Network-inclusive correlation rules from Section 12.2.
* Limit Network use to unclassified Network exposure relationships.
* Validate that expected Network/specialized overlap is not treated as correlation by itself.
* Confirm no duplication of Host Configuration inference outputs.

### Phase D – Full M8 Regression and Reporting Validation

* Execute full pipeline validation.
* Confirm deterministic output ordering.
* Confirm Output / Reporting serializes correlation findings without semantic alteration.
* Update Test Results Summary.

## 12. Correlation Rule Catalog

All rules MUST operate on prior findings provided by the orchestrator, produce new `DiscoveryFinding` objects, preserve source traceability, and avoid modifying upstream findings.

### 12.1 Specialized-Domain Correlation Rules

Rules SHALL be evaluated in this order:

1. COR-ID-AS-TL-001 — Identity, Application / Service, and Telemetry / Logging Correlation
2. COR-ID-AS-001 — Identity and Application / Service Correlation
3. COR-ID-TL-001 — Identity and Telemetry / Logging Correlation
4. COR-AS-TL-001 — Application / Service and Telemetry / Logging Correlation

The tri-domain rule has precedence. When `COR-ID-AS-TL-001` emits for a target, pairwise specialized-domain rules MUST NOT emit for that same target during the same run.

#### COR-ID-AS-TL-001

* **Category:** `identity_application_telemetry_correlation`
* **Trigger:** One or more Identity, one or more Application / Service, and one or more Telemetry / Logging exposure findings exist for the same execution target.
* **Correlated Domains:** `identity`, `application_service`, `telemetry_logging`
* **Rationale:** Target exposes identity, application/service, and telemetry/logging interfaces through independently governed discovery findings.

#### COR-ID-AS-001

* **Category:** `identity_application_correlation`
* **Trigger:** One or more Identity and one or more Application / Service exposure findings exist for the same execution target, and the tri-domain rule did not emit for that target.
* **Correlated Domains:** `identity`, `application_service`
* **Rationale:** Target exposes identity and application/service interfaces through independently governed discovery findings.

#### COR-ID-TL-001

* **Category:** `identity_telemetry_correlation`
* **Trigger:** One or more Identity and one or more Telemetry / Logging exposure findings exist for the same execution target, and the tri-domain rule did not emit for that target.
* **Correlated Domains:** `identity`, `telemetry_logging`
* **Rationale:** Target exposes identity and telemetry/logging interfaces through independently governed discovery findings.

#### COR-AS-TL-001

* **Category:** `application_telemetry_correlation`
* **Trigger:** One or more Application / Service and one or more Telemetry / Logging exposure findings exist for the same execution target, and the tri-domain rule did not emit for that target.
* **Correlated Domains:** `application_service`, `telemetry_logging`
* **Rationale:** Target exposes application/service and telemetry/logging interfaces through independently governed discovery findings.

### 12.2 Network-Inclusive Correlation Rules

Network-inclusive rules SHALL be implemented only after specialized-domain correlation behavior is validated.

These rules use Network findings only where a Network `open_port` finding represents unclassified Network exposure for the target. A Network finding is considered unclassified for correlation purposes only when no authorized specialized discovery finding exists for the same port-qualified target.

Network-inclusive rules MUST NOT treat expected overlap between Network `open_port` and specialized discovery findings as correlation by itself.

Rules SHALL be evaluated in this order after specialized-domain rules:

5. COR-NET-ID-001 — Unclassified Network and Identity Correlation
6. COR-NET-AS-001 — Unclassified Network and Application / Service Correlation
7. COR-NET-TL-001 — Unclassified Network and Telemetry / Logging Correlation
8. COR-NET-MULTI-001 — Unclassified Network and Multi-Domain Correlation

#### COR-NET-ID-001

* **Category:** `unclassified_network_identity_correlation`
* **Trigger:** One or more unclassified Network `open_port` findings and one or more Identity exposure findings exist for the same execution target.
* **Correlated Domains:** `network`, `identity`
* **Rationale:** Target exposes unclassified Network service exposure alongside identity-domain exposure.

#### COR-NET-AS-001

* **Category:** `unclassified_network_application_correlation`
* **Trigger:** One or more unclassified Network `open_port` findings and one or more Application / Service exposure findings exist for the same execution target.
* **Correlated Domains:** `network`, `application_service`
* **Rationale:** Target exposes unclassified Network service exposure alongside application/service exposure.

#### COR-NET-TL-001

* **Category:** `unclassified_network_telemetry_correlation`
* **Trigger:** One or more unclassified Network `open_port` findings and one or more Telemetry / Logging exposure findings exist for the same execution target.
* **Correlated Domains:** `network`, `telemetry_logging`
* **Rationale:** Target exposes unclassified Network service exposure alongside telemetry/logging exposure.

#### COR-NET-MULTI-001

* **Category:** `unclassified_network_multi_domain_correlation`
* **Trigger:** One or more unclassified Network `open_port` findings and exposure findings from two or more specialized semantic domains exist for the same execution target.
* **Correlated Domains:** `network` plus the applicable specialized domains
* **Rationale:** Target exposes unclassified Network service exposure alongside multiple independently governed specialized-domain exposure signals.

## 13. Finding Model

All Correlation Domain findings MUST conform to the `DiscoveryFinding` schema.

Required fields:

* `domain = "correlation"`
* `category = <approved correlation category>`
* `target = <execution target>`
* `evidence = <correlation evidence object>`
* `observed_at = <UTC timestamp>`

Required evidence structure:

```json
{
  "correlation_rule_id": "<approved-rule-id>",
  "correlated_domains": ["<domain-a>", "<domain-b>", "<optional-domain-c>"],
  "source_findings": [
    {
      "domain": "<source-domain>",
      "category": "<source-category>",
      "target": "<source-target>",
      "evidence": {},
      "observed_at": "<UTC timestamp>"
    }
  ],
  "rationale": "<approved bounded rationale>"
}
```

Correlation findings MUST NOT replace, suppress, mutate, reclassify, or override source findings.

---

## 14. Processing Controls

To maintain deterministic and auditable behavior:

* Rules MUST execute in fixed order.
* Duplicate correlation generation MUST be prevented.
* Output ordering MUST remain consistent.
* Identical input must always produce identical output, except for new UTC observation timestamps.
* Only authorized upstream categories and governed evidence fields may participate in rule evaluation.
* Governed evidence handling must remain exact-match, read-only, and specification-bound.
* No dynamic rule creation, runtime prioritization, randomization, concurrency, heuristic remapping, external enrichment, scoring, or adaptive analysis is permitted.

---

## 15. Failure and Error Handling

The Correlation Domain MUST handle errors gracefully.

Failure in one correlation rule MUST NOT prevent evaluation of subsequent approved rules unless the failure prevents safe continuation of the module.

Error handling MUST be structured, deterministic, non-sensitive, and must preserve upstream finding integrity.

Operational diagnostic output MUST NOT be interpreted as a correlation finding unless explicitly defined by this specification.

---

## 16. Acceptance Criteria

The Correlation Domain will be considered complete for M8 when:

* `Correlation_Domain.py` exists and implements the required module interface.
* The module executes after Host Configuration in approved runtime order.
* The domain consumes only orchestrator-provided prior findings.
* No direct network interaction, socket usage, probing, protocol interaction, authentication, or external enrichment occurs.
* Upstream findings remain read-only and unmodified in place.
* Specialized-domain rules execute in approved order with tri-domain precedence.
* Network-inclusive rules, when implemented, use only unclassified Network exposure relationships and do not treat default Network/specialized overlap as correlation.
* Outputs conform to the `DiscoveryFinding` schema.
* Outputs use `domain = "correlation"` and approved categories.
* Every output includes `correlation_rule_id`, `correlated_domains`, `source_findings`, and `rationale` evidence.
* Output is deterministic under identical input conditions, except for new UTC observation timestamps.
* No Host Configuration inference logic or category is duplicated.
* No regression defects are introduced in upstream, downstream, foundational, control-layer, or output-layer components.
* Test Results Summary is updated and approved.

---

## 17. Regression Requirements

Regression validation must confirm:

* Correlation consumes only authorized upstream categories and governed evidence fields.
* Upstream findings remain read-only and unmodified in place.
* No hidden coupling is introduced between producing discovery domains, Host Configuration, and Correlation Domain logic.
* Governed evidence fields are handled by exact-match, specification-bound logic only.
* Specialized-domain test cases cover Identity + Application, Identity + Telemetry, Application + Telemetry, and Identity + Application + Telemetry.
* The tri-domain case emits the tri-domain finding without redundant pairwise findings for the same target.
* Network-inclusive test cases confirm unclassified Network exposure handling without default overlap correlation.
* Output / Reporting remains presentation-only and does not define correlation semantics.

All regression testing must be documented in the Test Results Summary.

## 18. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Correlation duplicates Host Configuration inference | High | Prohibit Host inference recreation and Host category duplication |
| Network overlap is mislabeled as correlation | High | Limit Network use to rule-defined unclassified exposure relationships |
| Circular correlation through Host findings | High | Require source traceability and prohibit Host-to-source recursion |
| Tri-domain correlation is missed or duplicated through pairwise-only logic | Medium | Enforce explicit tri-domain precedence and regression validation |
| Upstream findings are modified in place | High | Enforce read-only, append-only input handling |
| Producer-owned evidence semantics are reinterpreted | High | Enforce exact-match, specification-bound evidence handling |
| Non-deterministic rule behavior | High | Enforce fixed rule order and static rule definitions |
| Business logic leaks into Core, CLI, Output, or other domains | High | Validate layer boundaries during implementation and regression testing |
