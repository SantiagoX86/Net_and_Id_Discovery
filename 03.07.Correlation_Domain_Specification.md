## 1. Purpose

The Correlation Domain defines the governed Milestone M8 correlation capability for the Enterprise Security Discovery Orchestration Framework.

This domain is responsible for deterministic, traceable cross-domain relationship analysis across previously produced governed findings. It operates only on existing findings produced by authorized discovery and inference capabilities and produces correlation-owned outputs without modifying, suppressing, merging, deduplicating, normalizing, reclassifying, or reinterpreting upstream findings in place.

The Correlation Domain exists to identify governed relationships between independently produced semantic signals while preserving producer-owned finding semantics, source traceability, deterministic execution, and enterprise SDLC control boundaries.

This specification defines the full intended Correlation Domain capability. Initial implementation is intentionally phased and limited to an introductory controlled rule set. Later M8 phases may expand toward the full authorized correlation model through controlled updates to the rule matrix, implementation alignment, and validation, without changing the architectural identity or responsibility boundary of this domain.

## 2. Architectural Boundary Statement

The Correlation Domain is a correlation-only consuming capability.

This domain consumes existing governed findings as read-only, append-only, contract-bound input and applies deterministic, specification-defined correlation rules to identify approved cross-domain relationships.

The Correlation Domain MUST NOT perform discovery, inference reserved for Host Configuration, direct network interaction, probing, socket usage, protocol interaction, authentication, enumeration, response parsing, banner grabbing, TLS handshakes, or target state modification.

The Correlation Domain MUST NOT mutate, overwrite, suppress, merge, deduplicate, normalize, reclassify, or reinterpret upstream findings in place.

This domain is distinct from:

* Discovery domains, which produce direct-observation findings
* Host Configuration Discovery, which produces host-level inference findings
* Core Framework, which performs orchestration, sequencing, aggregation, and failure isolation
* CLI / Control Layer, which performs input handling and invocation only
* Output / Reporting, which performs serialization and presentation only

This boundary is authoritative for all implementation, validation, downstream consumption, and future expansion of the Correlation Domain.

## 3. Vantage Point and Processing Assumptions

* No valid credentials are supplied or consumed by this domain.
* No direct network interaction is performed by this domain.
* No sockets are opened by this domain.
* No ports are probed by this domain.
* No protocol negotiation, authentication, banner collection, handshake inspection, response reading, metadata extraction, enumeration, or target interrogation is performed.
* All inputs are previously produced governed findings provided through the orchestrator-approved module contract.
* Upstream findings are treated as read-only, append-only, contract-bound, and authoritative within their producing or deriving contract.
* No upstream finding is modified, suppressed, merged, deduplicated, normalized, reclassified, or reinterpreted in place.
* Correlation output is derived only through deterministic, specification-defined rules.
* Observations are limited to approved relationships among existing governed findings.

## 4. Scope

### 4.1 In Scope

The following activities are in scope for the Correlation Domain:

* Consumption of previously produced governed findings through approved orchestrator-provided inputs
* Deterministic cross-domain relationship analysis across authorized governed findings
* Identification of approved relationships between independent governed semantic domains
* Generation of correlation-owned output findings using the normalized `DiscoveryFinding` schema during the initial implementation phase
* Preservation of source traceability for every correlation output
* Use of explicit, governed correlation rules with fixed rule identifiers and deterministic execution order
* Future controlled expansion of correlation rules to support the full intended Correlation Domain capability, including approved Network-derived correlation use cases where explicitly defined by this specification

### 4.2 Initial Implementation Scope

The introductory implementation phase is limited to deterministic correlation among independently governed specialized discovery-domain findings from:

* Identity Discovery Domain
* Application / Service Discovery Domain
* Telemetry / Logging Exposure Discovery Domain

The initial rule set is intentionally limited to:

* Identity and Application / Service exposure coexistence
* Identity and Telemetry / Logging exposure coexistence
* Application / Service and Telemetry / Logging exposure coexistence

This initial implementation scope is not the final intended Correlation Domain capability. It is the first controlled development phase used to validate correlation-only processing, source traceability, deterministic output, and layer-boundary enforcement before broader Correlation Domain functionality is implemented.

### 4.3 Planned Full-State Capability

The full intended Correlation Domain capability includes deterministic, traceable correlation across authorized governed findings from multiple producing and deriving domains, including future approved Network-based correlation use cases.

Network-derived correlation is expected to be introduced after the initial specialized-domain correlation behavior is implemented and validated. Future Network inclusion MUST be rule-defined and MUST NOT treat expected Network overlap with specialized discovery findings as a correlation insight by itself.

Future Network correlation use cases may include governed relationships involving unclassified Network exposure where such behavior is explicitly defined by approved Correlation Domain rules and does not duplicate Host Configuration inference responsibility.

The full-state Correlation Domain may also support additional correlation output structures beyond the initial `DiscoveryFinding` model only after controlled specification update, implementation alignment, and validation. Until such expansion is approved, initial implementation output is limited to normalized `DiscoveryFinding` records.

### 4.4 Out of Scope

The following activities are out of scope for the Correlation Domain:

* Direct discovery activity
* Direct network interaction
* Socket usage
* Port probing
* Protocol negotiation
* Authentication or credential usage
* Banner grabbing
* TLS handshake initiation or inspection
* Response reading or parsing
* Metadata extraction
* Enumeration
* Target state modification
* Host Configuration inference rule duplication
* Reclassification of source findings in place
* Suppression of upstream findings
* Deduplication, merging, or semantic reconciliation of upstream findings
* Expected Network and specialized-domain overlap correlation by default
* Risk scoring, severity scoring, prioritization, or exploitability assessment unless introduced through a future governed update
* Attack path generation unless introduced through a future governed update
* External data enrichment unless introduced through a future governed update
* Introduction of correlation logic into Core Framework, discovery domains, Host Configuration, CLI / Control Layer, Output / Reporting, Interface Layer, or Deployment Layer

## 5. Relationship to Other Domains and Layers

### 5.1 Network Discovery Domain

Network Discovery remains the authoritative producer of:

* `network/host_presence`
* `network/open_port`
* Generic bounded network exposure signals
* Network-owned `service_hint` values within its producing contract

The Correlation Domain MUST NOT reinterpret Network Discovery output as specialized application, identity, telemetry, host posture, protocol correctness, service health, or product meaning.

The Correlation Domain MUST NOT treat expected overlap between Network `open_port` findings and specialized discovery-domain findings as a correlation insight by itself.

Network findings are excluded from the initial Correlation Domain rule set. Network inclusion is planned for later Correlation Domain functionality where explicitly defined by governed correlation rules, including but not limited to future rule-defined handling of unclassified Network exposure relationships.

### 5.2 Identity Discovery Domain

Identity Discovery remains the authoritative producer of identity-related direct-observation findings and the authoritative owner of `identity_service_hint`.

The Correlation Domain may consume Identity Discovery findings only where explicitly authorized by this specification and approved correlation rules.

The Correlation Domain MUST NOT reuse, redefine, normalize, alias, extend, or reinterpret `identity_service_hint` values outside approved correlation-rule logic.

### 5.3 Application / Service Discovery Domain

Application / Service Discovery remains the authoritative producer of bounded application/service direct-observation findings and the authoritative owner of `application_service_hint`.

The Correlation Domain may consume Application / Service findings only where explicitly authorized by this specification and approved correlation rules.

The Correlation Domain MUST NOT reuse, redefine, normalize, alias, extend, or reinterpret `application_service_hint` values outside approved correlation-rule logic.

### 5.4 Telemetry / Logging Exposure Discovery Domain

Telemetry / Logging Exposure Discovery remains the authoritative producer of bounded telemetry/logging direct-observation findings and the authoritative owner of `telemetry_logging_hint`.

The Correlation Domain may consume Telemetry / Logging findings only where explicitly authorized by this specification and approved correlation rules.

The Correlation Domain MUST NOT reuse, redefine, normalize, alias, extend, or reinterpret `telemetry_logging_hint` values outside approved correlation-rule logic.

### 5.5 Host Configuration Discovery Domain

Host Configuration Discovery remains the authorized inference-only downstream consumer domain for host-level posture inference.

The Correlation Domain may ingest Host Configuration findings as contextual inputs in later approved phases. However, Host Configuration findings MUST NOT be treated as independent correlation signals when a proposed correlation merely relates the Host Configuration finding back to one of its own source domains.

The Correlation Domain MUST NOT recreate Host Configuration inference rules, duplicate Host Configuration output categories, or emit correlation findings for relationships already represented by approved Host Configuration inference outputs.

Where Host Configuration findings are consumed in future approved correlation rules, the rule MUST preserve source-finding traceability and MUST avoid circular or redundant correlation.

Host Configuration findings are excluded from the initial Correlation Domain rule set.

### 5.6 Core Framework / Orchestration Layer

The Core Framework remains responsible only for orchestration, sequencing, aggregation, approved input propagation, and failure isolation.

The Correlation Domain may be registered as an orchestrated module after Host Configuration Discovery in deterministic runtime order.

The Core Framework MUST NOT perform correlation logic, interpret correlation rule semantics, mutate upstream findings for correlation, suppress upstream findings, or generate correlation outputs on behalf of this domain.

### 5.7 CLI / Control Layer

The CLI / Control Layer remains responsible only for controlled input handling, execution context construction, and orchestrator invocation.

The CLI MUST NOT contain correlation rules, correlation selection logic, finding reinterpretation logic, or domain-specific correlation behavior.

Initial Correlation Domain integration with the CLI is limited to module registration in the approved deterministic runtime sequence.

### 5.8 Output / Reporting Layer

Output / Reporting remains presentation-only.

The Correlation Domain produces governed outputs. Output / Reporting may serialize or present those outputs, but it does not define, expand, suppress, merge, deduplicate, prioritize, score, normalize, or reinterpret Correlation Domain semantics.

Markdown sections, headings, tables, summaries, or grouping behavior MUST NOT be treated as the technical source of truth for correlation semantics.

The structured `DiscoveryFinding` output remains the authoritative technical source of truth for initial Correlation Domain outputs.

## 6. Technical Approach

The Correlation Domain extends the existing Core Framework module contract as a correlation-only consuming module.

The domain receives prior findings from the Discovery Orchestrator and applies deterministic, specification-defined correlation rules against authorized upstream findings. Correlation rules evaluate relationships among governed findings without modifying source findings.

Initial implementation emits new normalized `DiscoveryFinding` records under the Correlation Domain output contract. These outputs are append-only and preserve traceability to all source findings used by the rule.

All correlation processing MUST be deterministic. Identical input findings MUST produce identical correlation outputs in identical ordering, except for UTC observation timestamps generated for new correlation findings.

No rule may be dynamically generated at runtime. No rule may be reordered, prioritized, skipped, expanded, or modified based on environmental conditions unless explicitly authorized by this specification.

The domain MUST use only local in-memory processing of orchestrator-provided findings. It MUST NOT access external systems, perform network interaction, retrieve enrichment data, or depend on output formatting behavior.

## 7. Runtime Position and Execution Ordering

The Correlation Domain is a downstream consuming module and is expected to execute after discovery producers and Host Configuration inference.

For the approved M8 runtime model, the deterministic execution ordering is:

1. Network Discovery Domain
2. Identity Discovery Domain
3. Application / Service Discovery Domain
4. Telemetry / Logging Exposure Discovery Domain
5. Host Configuration Discovery Domain
6. Correlation Domain

This ordering allows the Correlation Domain to receive the full governed finding set produced by previously executed modules while preserving strict layer responsibility boundaries.

The Correlation Domain MUST NOT use runtime position to reinterpret, suppress, reorder, or mutate upstream findings.

## 8. Input Model

The Correlation Domain receives prior findings from the Discovery Orchestrator as a read-only, append-only collection.

* Prior findings represent aggregated outputs from previously executed modules.
* Input findings are immutable and must not be modified.
* Correlation logic operates exclusively on provided findings.
* Only explicitly authorized upstream categories and governed evidence fields may be consumed by approved correlation rules.
* Presence of additional upstream findings does not, by itself, authorize their use by this domain.

### 8.1 Initial Authorized Upstream Categories

For the introductory implementation phase, the Correlation Domain is authorized to consume the following upstream finding categories:

* `domain = "identity"`, `category = "identity_service_exposed"`
* `domain = "application_service"`, `category = "application_service_exposed"`
* `domain = "telemetry_logging"`, `category = "telemetry_logging_exposed"`

### 8.2 Planned Future Upstream Category Expansion

The full intended Correlation Domain capability is expected to include additional governed upstream categories after initial behavior is validated.

Future expansion may include:

* Network Discovery findings, including approved use cases involving unclassified Network exposure relationships
* Host Configuration findings as bounded posture context where circular or redundant correlation is explicitly prevented
* Additional future-domain findings authorized through governed change control

No planned future input is authorized for implementation until it is explicitly defined in this specification, implemented through controlled change, and validated.

### 8.3 Initial Authorized Governed Evidence Fields

Where explicitly referenced by initial correlation rule logic, this domain is authorized to consume the following governed upstream evidence fields:

* `evidence.identity_service_hint`
* `evidence.application_service_hint`
* `evidence.telemetry_logging_hint`
* `evidence.port`
* `evidence.method`
* `evidence.timeout_s`

Producer-owned classification fields MUST be treated as:

* Read-only
* Authoritative within their producing contract
* Exact-match only
* Non-derivable within this domain
* Non-normalized unless explicitly authorized by this specification

### 8.4 Consuming Constraints

The Correlation Domain MUST NOT:

* Modify upstream findings in place
* Alias one governed evidence field to another
* Normalize, reinterpret, synonym-expand, or heuristically remap governed upstream evidence values
* Depend on undocumented producer behavior
* Treat expected Network and specialized-domain overlap as correlation by itself
* Correlate Host Configuration findings back to their own source findings as if that were independent evidence
* Consume future-domain outputs unless and until explicitly authorized by a future controlled specification update

For the introductory implementation phase, the Correlation Domain emits normalized `DiscoveryFinding` records only.

This is the approved initial output model and is intended to validate deterministic correlation processing, traceability, serialization compatibility, and layer-boundary enforcement using the existing framework output contract.

### 10.2 Planned Full-State Output Model

The final intended Correlation Domain capability may require additional correlation output structures beyond normalized `DiscoveryFinding` records.

Such expansion is not authorized by implication. Any future output model expansion requires:

* Controlled update to this specification
* Impact assessment across Core Framework, Output / Reporting, Interface Layer, and validation artifacts
* Implementation alignment
* Regression validation
* Documentation update and approval

Until such expansion is approved, `DiscoveryFinding` remains the sole authorized Correlation Domain output model.

### 10.3 Output Domain

All initial findings generated by this domain MUST use:

* `domain = "correlation"`

### 10.4 Initial Approved Output Categories

The following `DiscoveryFinding.category` values are authorized for the introductory implementation phase:

* `identity_application_correlation`
* `identity_telemetry_correlation`
* `application_telemetry_correlation`

These initial categories are not the full final category set for the Correlation Domain. They represent the first controlled development phase. Additional categories, including Network-inclusive categories, may be added through controlled update after initial functionality is validated.

No alternate category names, synonymous labels, or additional categories are authorized for implementation until this specification is updated.

### 10.5 Category Determination Constraints

* A single rule MUST NOT emit multiple categories.
* Category assignment MUST be deterministic and rule-driven.
* Category values MUST NOT be derived dynamically at runtime.
* Correlation categories MUST NOT duplicate Host Configuration inference categories.
* Correlation categories MUST NOT replace, rename, or suppress upstream categories.

## 11. Evidence Collection Strategy

Each Correlation Domain finding must include sufficient evidence to audit the correlation without re-executing the discovery pipeline.

Required evidence elements for initial correlation findings include:

* `correlation_rule_id`
* `correlated_domains`
* `source_findings`
* `rationale`

Optional controlled evidence elements may include:

* `correlation_type`
* `correlation_scope`
* `notes`

Evidence must be derived strictly from previously produced governed findings.

Evidence MUST NOT include:

* Response content
* Banners
* Headers
* Handshake artifacts
* Metadata extracted from target systems
* Certificate details
* Vendor or product identification not present in governed upstream findings
* Risk score or severity score unless authorized by future controlled update
* Attack path claims unless authorized by future controlled update
* Any enrichment from external systems unless authorized by future controlled update

### 11.1 Source Finding Traceability

The `source_findings` evidence element MUST preserve traceability to every source finding used by the rule.

Initial implementation may preserve source traceability using deterministic source-finding snapshots containing controlled fields such as:

* `domain`
* `category`
* `target`
* `evidence`
* `observed_at`

Source-finding snapshots MUST NOT alter the meaning of upstream findings.

Source-finding snapshots MUST NOT replace the upstream finding as the authoritative source of truth.

Where future framework support introduces stable finding identifiers, Correlation Domain evidence may use those identifiers in addition to or instead of snapshots, subject to controlled specification update.

## 12. Correlation Design Principles

All Correlation Domain logic must adhere to the following principles:

* **Correlation-only processing:** The domain identifies approved relationships among existing governed findings only.
* **No discovery activity:** The domain performs no direct observation and no network interaction.
* **No Host inference duplication:** The domain does not recreate Host Configuration rule logic or categories.
* **Independent semantic relationship requirement:** Initial correlation rules identify relationships between independently governed semantic domains.
* **No default Network overlap correlation:** Expected Network/specialized-domain overlap is not a correlation insight by itself.
* **Read-only upstream handling:** Upstream findings are never modified in place.
* **Append-only output:** The domain emits new correlation-owned findings only.
* **Deterministic execution:** Rules execute in fixed order and produce consistent output for identical input findings.
* **Traceable evidence:** Every correlation finding identifies the source findings used by the rule.
* **Producer-owned semantics:** Upstream domain, category, and evidence meanings remain owned by the producing or deriving domain.

### 12.1 Correlation Boundary Principle

M8 correlation rules MUST identify deterministic, traceable relationships between independent governed semantic domains.

M8 MUST NOT treat a derived Host Configuration finding as an independent correlation signal when the proposed correlation merely relates that Host Configuration finding back to one of its own source domains.

M8 MUST NOT treat expected overlap between Network Discovery `open_port` findings and specialized discovery-domain findings as a correlation insight by itself.

M8 MUST NOT recreate Host Configuration inference rules, duplicate Host Configuration output categories, or emit correlation findings for relationships already represented by approved Host Configuration inference outputs.

Where Host Configuration findings are consumed in future approved phases, they MAY be used only as bounded posture context and only where source-finding traceability confirms the rule does not create circular or redundant correlation.

### 12.2 Initial Rule Philosophy

Initial Correlation Domain rules identify coexistence of independently governed specialized-domain exposure findings on the same execution target.

This initial philosophy is intentionally limited to the first development phase and is not the final limit of Correlation Domain functionality.

The final intended Correlation Domain capability may include additional rule families, Network-derived correlation use cases, Host-contextual correlation, richer relationship typing, and expanded output structures where explicitly defined by controlled specification update and validated.

## 13. Deterministic Rule Definition Matrix

The following rules define the approved introductory correlation logic for the Correlation Domain.

All initial correlation rules:

* Operate on prior findings provided by the orchestrator
* Produce new `DiscoveryFinding` objects
* Must not modify input findings
* Must execute in the order defined below
* Must emit at most one finding per rule per target
* Must consume only explicitly authorized upstream categories and governed evidence fields
* Must apply exact-match, deterministic, specification-defined logic only
* Must not rely on undocumented producer behavior, incidental formatting, or output presentation behavior

Additional future correlation rules are planned but are not authorized for implementation until added to this specification through controlled update.

### 13.1 Rule Execution Order

Rules SHALL be evaluated in the following exact sequence:

1. COR-ID-AS-001 — Identity and Application / Service Correlation
2. COR-ID-TL-001 — Identity and Telemetry / Logging Correlation
3. COR-AS-TL-001 — Application / Service and Telemetry / Logging Correlation

Output findings MUST be emitted in this same order.

### 13.2 Rule Definitions

#### Rule ID: COR-ID-AS-001

* **Category:** `identity_application_correlation`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
  * AND one or more prior findings exist where:
    * `domain = "application_service"`
    * `category = "application_service_exposed"`
  * AND the findings apply to the same execution target.
* **Correlated Domains:**
  * `identity`
  * `application_service`
* **Rationale:** Target exposes both identity-related and application/service interfaces through independently governed discovery findings.
* **Constraints:**
  * MUST NOT require or consume Network `open_port` findings.
  * MUST NOT derive Host Configuration posture.
  * MUST NOT imply protocol correctness, authentication state, service health, product identity, or application-layer behavior.

#### Rule ID: COR-ID-TL-001

* **Category:** `identity_telemetry_correlation`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
  * AND one or more prior findings exist where:
    * `domain = "telemetry_logging"`
    * `category = "telemetry_logging_exposed"`
  * AND the findings apply to the same execution target.
* **Correlated Domains:**
  * `identity`
  * `telemetry_logging`
* **Rationale:** Target exposes both identity-related and telemetry/logging interfaces through independently governed discovery findings.
* **Constraints:**
  * MUST NOT require or consume Network `open_port` findings.
  * MUST NOT derive Host Configuration posture.
  * MUST NOT imply protocol correctness, authentication state, service health, ingestion health, product identity, or telemetry pipeline behavior.

#### Rule ID: COR-AS-TL-001

* **Category:** `application_telemetry_correlation`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "application_service"`
    * `category = "application_service_exposed"`
  * AND one or more prior findings exist where:
    * `domain = "telemetry_logging"`
    * `category = "telemetry_logging_exposed"`
  * AND the findings apply to the same execution target.
* **Correlated Domains:**
  * `application_service`
  * `telemetry_logging`
* **Rationale:** Target exposes both application/service and telemetry/logging interfaces through independently governed discovery findings.
* **Constraints:**
  * MUST NOT require or consume Network `open_port` findings.
  * MUST NOT derive Host Configuration posture.
  * MUST NOT imply protocol correctness, service health, ingestion health, product identity, application-layer behavior, or telemetry pipeline behavior.

## 17. Failure and Error Handling

The Correlation Domain MUST handle errors gracefully.

Failure in one correlation rule MUST NOT prevent evaluation of subsequent approved rules unless the failure prevents safe continuation of the module.

Error handling MUST:

* Be structured
* Be deterministic
* Avoid sensitive data collection
* Avoid modification of upstream findings
* Preserve failure isolation
* Support auditability

Any structured correlation-domain error output MUST remain operational diagnostic information and MUST NOT be interpreted as a correlation finding unless explicitly defined by this specification.

|------|--------|------------|
| Correlation duplicates Host Configuration inference | High | Explicitly prohibit Host inference recreation and Host category duplication |
| Network overlap is mislabeled as correlation | High | Exclude Network from initial rules and require future Network rules to avoid default overlap correlation |
| Circular correlation through Host findings | High | Require source traceability and prohibit Host-to-source recursion |
| Upstream findings are modified in place | High | Enforce read-only, append-only input handling and regression validation |
| Producer-owned evidence semantics are reinterpreted | High | Enforce exact-match, specification-bound governed evidence handling |
| Non-deterministic rule behavior | High | Enforce fixed rule order and static rule definitions |
| Scope creep into risk scoring or attack-path generation | Medium | Defer such functionality unless introduced through controlled update |
| Business logic leaks into Core, CLI, or Output | High | Validate layer boundaries during implementation and regression testing |
| Output model expands prematurely | Medium | Limit initial implementation to `DiscoveryFinding` records and require controlled update for future output structures |
| Specification underrepresents full intended M8 scope | Medium | Document initial implementation separately from planned full-state Correlation Domain capability |

## 22. Governance and Change Control

No modifications to previously validated components are permitted without:

* Formal change documentation
* Impact assessment across affected layers and domains
* Re-execution of required regression tests
* Version increment and updated traceability where applicable

All changes introduced under the Correlation Domain must remain isolated unless formally authorized.

Any expansion of authorized upstream categories, governed evidence dependencies, output categories, rule behavior, Network-inclusive correlation, Host-contextual correlation, scoring, attack-path analysis, external enrichment, or correlation output structures requires controlled specification update and alignment across implementation and validation artifacts.

All Correlation Domain implementation and validation activities must preserve:

* Deterministic behavior
* Producer-owned semantics
* Read-only upstream finding handling
* Append-only output generation
* Source traceability
* Domain isolation
* Layer-responsibility boundaries
* SDLC traceability to approved governing documents
