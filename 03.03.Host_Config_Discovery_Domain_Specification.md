## 1. Purpose

The Host Configuration Discovery Domain defines a deterministic, inference-only domain responsible for deriving host-level security posture insights based solely on previously observed discovery findings.

This domain operates without direct interaction with target systems and produces normalized, auditable findings aligned with the Core Framework schema.

---

## 2. Vantage Point Assumptions

* No valid credentials are supplied.
* No direct network interaction is performed by this domain.
* All inputs are derived from prior discovery domains (Network and Identity).
* No modification of target systems or upstream findings occurs.
* Observations are limited to inference from externally observable signals.

---

## 3. Scope

### 3.1 In Scope

* Consumption of findings from:
  * Network Discovery Domain
  * Identity Discovery Domain
* Deterministic, rule-based inference of host configuration posture
* Generation of normalized findings using the DiscoveryFinding schema
* Identification of security-relevant configuration indicators based on exposed services

### 3.2 Out of Scope

* Any form of direct network interaction
* Port probing or connectivity testing
* Protocol negotiation or enumeration
* Authentication or credential usage
* Modification of upstream findings
* Cross-domain logic coupling
* Any modification to frozen M1–M5 components without formal change control

---

## 4. Technical Approach

The Host Configuration Discovery Domain extends the Core Framework module contract as an inference-only domain. This domain relies on orchestrator-provided prior findings, enabled by a controlled Core Framework (M1) contract extension. No additional orchestration changes are required beyond this defined mechanism, and no changes are introduced to existing discovery domains.

This domain consumes aggregated findings provided by the Discovery Orchestrator and applies deterministic, rule-based evaluation to derive host configuration insights. No network operations, socket interactions, or protocol-level activity are performed.

Inference logic evaluates the presence and combination of previously observed service exposures to identify configuration patterns and potential security posture indicators. All outputs are newly generated findings and do not modify or overwrite upstream data.

All processing must remain deterministic. Identical input findings must produce identical output findings. No randomness, concurrency, or time-dependent behavior is permitted.

---

## 5. Input Model

The domain receives prior discovery findings from the Discovery Orchestrator as a read-only, append-only collection.

- Prior findings represent aggregated outputs from previously executed modules
- Input data is immutable and must not be modified
- All inference logic operates exclusively on provided findings

## 11. Inference Strategy

The Host Configuration Discovery Domain applies a deterministic, rule-based methodology to derive host posture insights from previously observed service exposure.

### 11.1 Inference Design Principles

All inference logic must adhere to the following constraints:

* **Input-only processing:** Only consumes findings provided by the orchestrator
* **No network interaction:** No sockets, probes, or connectivity checks
* **Deterministic evaluation:** Fixed rule ordering with no randomness
* **Read-only input:** Upstream findings must not be modified
* **Traceable evidence:** All outputs must reference source findings

Inference execution SHALL follow a fixed, rule-based evaluation model:

* Rules are evaluated sequentially in a predefined, deterministic order
* Each rule independently evaluates the full set of prior findings
* Rules MUST NOT modify shared state or intermediate results
* No implicit or dynamic inter-rule dependencies are permitted

The inference engine SHALL:

* Consume all prior findings as a read-only dataset
* Apply all rules consistently for a given execution target
* Emit only normalized `DiscoveryFinding` objects
* Produce identical outputs when provided identical inputs

No dynamic rule creation, runtime prioritization, randomized processing, or conditional reordering of rule execution is permitted.

All inference behavior implemented in code MUST be explicitly defined within this specification prior to implementation.

### 11.2 Inference Categories

All derived findings produced by the Host Configuration Discovery Domain SHALL conform to a strictly controlled output model.

#### 11.2.1 Output Domain

All findings generated by this domain MUST use:

* `domain = "host_configuration"`

#### 11.2.2 Approved Output Categories

The following `DiscoveryFinding.category` values are authorized for initial M6 implementation:

* `remote_management_exposure`
* `directory_service_exposure`
* `smb_exposure_posture`
* `rpc_exposure_indicator`
* `unclassified_network_service_exposure`
* `reachable_host_without_identity_exposure`

No additional categories, alternate naming, or synonymous labels are permitted without formal specification update.

#### 11.2.3 Category Determination Constraints

* Each rule MUST map to exactly one approved category
* A single rule MUST NOT emit multiple categories
* Category assignment MUST be deterministic and rule-driven
* Category values MUST NOT be derived dynamically at runtime

#### 11.2.4 Conceptual Mapping (Non-Executable Reference)

The following mappings define how upstream identity service exposure signals conceptually relate to host configuration inference categories.

This subsection is **descriptive only** and does not define execution logic.

##### Remote Management Exposure
* Associated with presence of:
  * WinRM (5985 / 5986)
  * RDP (3389)

##### Directory Service Exposure
* Associated with presence of:
  * LDAP (389)
  * LDAPS (636)
  * Kerberos (88)

##### SMB Exposure Posture
* Associated with presence of:
  * SMB (445)
  * NetBIOS (139)

##### RPC Exposure Indicator
* Associated with presence of:
  * RPC (135)

### 11.3 Evidence Collection Strategy

Each inferred finding must include:

* Source findings contributing to the inference
* Applied inference rule identifier
* Human-readable rationale explaining the conclusion

Evidence must be derived strictly from previously observed findings and must not introduce new external data.

### 11.4 Confidence Assignment

Confidence must be statically assigned based on rule strength and signal reliability.

* High (0.8–0.95): Strong indicators of host configuration state, including highly reliable single-service signals or multiple corroborating signals
* Medium (0.5–0.79): Moderate indicators based on single observable signals or partial context
* Low (<0.5): Weak or indirect signals with limited confidence

Confidence assignment MUST be deterministic and defined per rule.

No dynamic or probabilistic adjustment is permitted.

### 11.5 Processing Controls

To maintain enterprise-safe and deterministic behavior:

* Rules must execute in fixed order
* Duplicate inference generation must be prevented
* Output ordering must remain consistent
* Identical input must always produce identical output

#### Rule Execution Ordering

All inference rules SHALL be evaluated in a fixed, predefined order.

* Rules are evaluated sequentially (no parallel execution)
* Each rule evaluates against the full set of prior findings
* Rule execution order MUST remain consistent across all runs
* Output findings MUST be emitted in the same order as rule evaluation

No dynamic reordering, prioritization, or conditional skipping of rules is permitted unless explicitly defined in this specification.

This ensures:

* Deterministic output generation
* Consistent behavior across environments
* Full auditability of inference logic

### 11.6 Deterministic Rule Definition Matrix

The following rules define the complete, implementation-authoritative inference logic for the Host Configuration Discovery Domain.

All rules:

* Operate on prior findings provided by the orchestrator
* Produce new `DiscoveryFinding` objects
* Must not modify input findings
* Must execute in the order defined below
* Must emit at most one finding per rule per target

#### 11.6.1 Rule Execution Order

Rules SHALL be evaluated in the following exact sequence:

1. HC-RM-001 — Remote Management Exposure (WinRM)
2. HC-RM-002 — Remote Management Exposure (RDP)
3. HC-DS-001 — Directory Service Exposure
4. HC-SMB-001 — SMB Exposure Posture
5. HC-RPC-001 — RPC Exposure Indicator
6. HC-NET-001 — Unclassified Network Service Exposure
7. HC-NET-002 — Reachable Host Without Identity Service Exposure

Output findings MUST be emitted in this same order.

#### 11.6.2 Rule Definitions

##### Rule ID: HC-RM-001
* **Category:** `remote_management_exposure`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `evidence.service_hint` is `WinRM` or `WinRM-HTTPS`
* **Confidence:** `0.85`
* **Rationale:** Host exposes remote management capability via WinRM-related service exposure

##### Rule ID: HC-RM-002
* **Category:** `remote_management_exposure`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `evidence.service_hint` is `RDP`
* **Confidence:** `0.85`
* **Rationale:** Host exposes remote desktop capability

##### Rule ID: HC-DS-001
* **Category:** `directory_service_exposure`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `evidence.service_hint` is `LDAP`, `LDAPS`, or `Kerberos`
* **Confidence:** `0.90`
* **Rationale:** Host presents directory-service-related exposure signals

##### Rule ID: HC-SMB-001
* **Category:** `smb_exposure_posture`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `evidence.service_hint` is `SMB` or `NetBIOS-SSN`
* **Confidence:** `0.80`
* **Rationale:** Host exposes SMB-related service surface

##### Rule ID: HC-RPC-001
* **Category:** `rpc_exposure_indicator`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `evidence.service_hint` is `RPC`
* **Confidence:** `0.75`
* **Rationale:** Host exposes RPC endpoint mapper service

##### Rule ID: HC-NET-001
* **Category:** `unclassified_network_service_exposure`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "network"`
    * `category = "open_port"`
  * AND no prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
    * `target` matches the same port-qualified target as the triggering network finding

Target matching for this rule MUST be performed using the full port-qualified target (e.g., `192.168.1.10:445`) to ensure accurate correlation between network and identity findings.

This requirement applies only to rule evaluation logic. Output normalization rules defined in Section 11.6.4 remain unchanged.
* **Confidence:** `0.60`
* **Rationale:** Host exposes a reachable network service that is not classified by the current Identity Discovery rule set

##### Rule ID: HC-NET-002
* **Category:** `reachable_host_without_identity_exposure`
* **Trigger Condition:**
  * One or more prior findings exist where:
    * `domain = "network"`
    * `category = "host_presence"`
    * `evidence.reachable` is `true`
  * AND no prior findings exist where:
    * `domain = "identity"`
    * `category = "identity_service_exposed"`
* **Confidence:** `0.55`
* **Rationale:** Host is reachable from the discovery vantage point, but no identity-related service exposure was observed

#### 11.6.3 Evidence Requirements

Each derived finding MUST include:

* `rule_id`
* `source_categories`
* `source_targets`
* `source_ports`
* `rationale`

All evidence fields MUST be represented as deterministic, de-duplicated lists where applicable, and must maintain consistent ordering across executions.

#### 11.6.4 Target Normalization

Derived findings MUST use the base host target (no port).

Example:

* Input: `192.168.145.129:5985`
* Output: `192.168.145.129`

#### 11.6.5 Duplicate Prevention

Each rule may emit only one finding per execution target.

If multiple upstream findings satisfy a rule:

* All may be referenced in evidence
* Only one derived finding is produced

Deduplication applies only within an individual rule. Cross-rule consolidation or suppression of findings is not permitted.
