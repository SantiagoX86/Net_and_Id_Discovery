## 1. Purpose

The Network Discovery Domain defines a deterministic, discovery-only domain responsible for producing generic network reachability and generic TCP exposure findings from an approved discovery vantage point.

This domain operates in an agentless, non-exploitative manner and produces normalized `DiscoveryFinding` outputs aligned with the Core Framework schema.

Network Discovery is the authoritative generic producer of:

* `network/host_presence`
* `network/open_port`
* retained Network-owned probe error findings, where applicable

This domain exists to identify externally observable host presence and bounded generic TCP exposure without assigning application, identity, telemetry, posture, health, product, role, configuration, or service-validity meaning.

---

## 2. Architectural Boundary Statement

The Network Discovery Domain is a discovery-only producer of direct-observation findings.

Network Discovery owns generic TCP reachability semantics for `network/open_port` findings. A `network/open_port` finding represents only that a TCP connection was successfully established to an approved TCP port from the discovery vantage point.

Network Discovery MUST NOT absorb semantic ownership from specialized discovery domains.

Network Discovery MUST NOT determine:

* application behavior
* identity service correctness
* telemetry or logging ingestion behavior
* host configuration posture
* product identity
* service health
* authentication state
* TLS state
* protocol correctness
* role or business function

Network Discovery MUST remain independent of downstream inference, correlation, Output / Reporting presentation behavior, and specialized-domain interpretation.

---

## 3. Vantage Point Assumptions

* Discovery is executed from an authorized external or minimally privileged network vantage point.
* No valid credentials are supplied.
* No agents are deployed to target systems.
* No exploitation, brute force, authentication bypass, or attack simulation techniques are used.
* Observations are limited to externally observable host presence and TCP connectivity signals.
* TCP observations are limited to the approved project-scoped TCP port inventory defined by the controlling baseline and this specification.
* TCP interaction is connect-only.
* No payloads are transmitted.
* No response content is read, parsed, or interpreted.
* No protocol negotiation, TLS handshake, banner grabbing, authentication, enumeration, or adaptive probing is performed.
* No upstream findings are consumed by this domain for meaning assignment.

## 4. Scope

### 4.1 In Scope

The following activities are in scope for Network Discovery:

* Host presence observation using approved low-impact network reachability methods.
* Generic TCP exposure observation through successful connect-only TCP establishment.
* Production of `network/host_presence` findings.
* Production of `network/open_port` findings across the approved project-scoped TCP port inventory.
* Deterministic static classification of observed TCP ports using the Network-owned `service_hint` evidence field.
* Structured handling of retained Network-owned probe errors, where applicable.
* Fixed-order probing of approved Network TCP ports.
* Normalized finding output using the `DiscoveryFinding` schema.

### 4.2 Out of Scope

The following activities are out of scope for Network Discovery:

* Application / Service Discovery semantic classification beyond Network-owned `service_hint`.
* Identity Discovery semantic classification beyond Network-owned `service_hint`.
* Telemetry / Logging Exposure Discovery semantic classification beyond Network-owned `service_hint`.
* Host configuration inference.
* Correlation or cross-domain relationship analysis.
* Protocol negotiation.
* Payload transmission.
* Socket reads.
* Response parsing.
* TLS handshake initiation or inspection.
* Banner grabbing.
* Authentication or credential usage.
* Enumeration.
* Product, vendor, version, or framework fingerprinting.
* Service health validation.
* Configuration posture interpretation.
* Adaptive or response-dependent probing.
* Suppression of valid Network findings because another authorized domain observes the same TCP port.
* Reclassification of specialized discovery findings as Network findings.
* Use of Output / Reporting presentation behavior to define Network discovery semantics.

## 5. Relationship to Existing Domains

### 5.1 Identity Discovery

Identity Discovery owns domain-specific direct-observation findings for approved identity-related exposure.

Network Discovery may independently produce `network/open_port` findings for identity-related TCP ports when those ports are included in the approved project-scoped TCP port inventory.

Network `service_hint` values for identity-overlap ports are generic Network-owned classifications only. They MUST NOT imply identity service correctness, authentication capability, directory availability, domain role, trust relationship, or protocol negotiation.

Identity Discovery MUST NOT emit `open_port`.

Network Discovery MUST NOT emit `identity_service_exposed`.

### 5.2 Application / Service Discovery

Application / Service Discovery owns domain-specific direct-observation findings for approved application/service interface exposure.

Network Discovery may independently produce `network/open_port` findings for application/service TCP ports when those ports are included in the approved project-scoped TCP port inventory.

Network `service_hint` values for application/service-overlap ports are generic Network-owned classifications only. They MUST NOT imply HTTP behavior, TLS behavior, application health, web server identity, framework identity, or successful application-layer interaction.

Application / Service Discovery MUST NOT emit `open_port`.

Network Discovery MUST NOT emit `application_service_exposed` or use `application_service_hint`.

### 5.3 Telemetry / Logging Exposure Discovery

Telemetry / Logging Exposure Discovery owns domain-specific direct-observation findings for approved telemetry/logging-related exposure.

Network Discovery may independently produce `network/open_port` findings for telemetry/logging TCP ports when those ports are included in the approved project-scoped TCP port inventory.

Network `service_hint` values for telemetry/logging-overlap ports are generic Network-owned classifications only. They MUST NOT imply ingestion health, log collection, observability maturity, collector identity, backend availability, telemetry pipeline behavior, or TLS validity.

Telemetry / Logging Exposure Discovery MUST NOT emit `open_port`.

Network Discovery MUST NOT emit `telemetry_logging_exposed` or use `telemetry_logging_hint`.

### 5.4 Host Configuration Discovery

Host Configuration Discovery is an inference-only downstream consumer domain.

Network Discovery MUST NOT perform host posture inference. Network Discovery MUST NOT consume upstream findings, correlate signals, or derive host-level conclusions.

Host Configuration may consume authorized Network findings only where explicitly permitted by its own governed specification.

Network Discovery output meaning does not depend on Host Configuration interpretation.

### 5.5 Output / Reporting

Output / Reporting is presentation-only.

Network Discovery produces governed findings. Output / Reporting may serialize or present those findings, but it does not define, expand, suppress, merge, deduplicate, or reinterpret Network semantics.

Markdown report sections, tables, headings, or grouping behavior MUST NOT be treated as the technical source of truth for whether Network exposure exists.

The structured `DiscoveryFinding` output remains the authoritative source of truth.

## 6. Technical Approach

Network Discovery performs two bounded forms of direct observation:

1. Host presence observation.
2. Generic TCP connect-only exposure observation.

Host presence observation produces `network/host_presence` findings. Under the current approved model, host presence is a host-level reachability observation and MUST NOT imply TCP service availability.

Generic TCP exposure observation is performed using a single bounded TCP connection attempt against each approved TCP port in the Network port inventory. A successful connection produces a `network/open_port` finding.

TCP probing MUST remain connect-only. The domain MUST terminate the connection immediately after the connection result is determined.

No application-layer data may be transmitted. No response data may be read. No protocol state may be negotiated. No target metadata may be extracted.

All Network classification is static and deterministic. The `service_hint` value is derived solely from:

* the destination TCP port
* successful TCP connection establishment
* the static Network-owned port-to-service mapping defined by this specification

Network Discovery does not validate that the expected service is actually running on the port. It records only the static Network classification associated with the port.

## 7. Runtime Position and Execution Ordering

Network Discovery is a discovery producer and is expected to execute before specialized discovery producers and downstream inference consumers.

For the current approved runtime model, the expected deterministic ordering is:

1. Network Discovery Domain
2. Identity Discovery Domain
3. Application / Service Discovery Domain
4. Telemetry / Logging Exposure Discovery Domain
5. Host Configuration Discovery Domain

This ordering preserves clarity of the discovery pipeline but does not authorize Network Discovery to consume downstream findings or adjust behavior based on later domain output.

Network Discovery MUST use only its local execution context and approved local configuration.

## 9. Probe Design Principles

All Network TCP probes MUST adhere to the following principles:

* **Connect-only interaction:** Probes are limited strictly to TCP connection establishment.
* **Single-attempt behavior:** Each approved port receives one bounded TCP connection attempt per run unless future retry behavior is explicitly approved.
* **Immediate termination:** The connection is closed immediately after the connection result is determined.
* **No payloads:** No application-layer data, protocol-specific bytes, credentials, headers, tokens, certificates, messages, requests, or payloads are transmitted.
* **No socket reads:** No response data is read from the target.
* **No protocol negotiation:** No SMB, LDAP, RPC, WinRM, RDP, HTTP, TLS, syslog, OTLP, GELF, or other protocol negotiation is performed.
* **Static timeout:** Timeout values are fixed and bounded.
* **Low-noise operation:** Probe behavior must remain predictable and limited.
* **Deterministic ordering:** Probe order must be fixed and reproducible.
* **Failure isolation:** Probe failures must not interrupt execution of other ports or other modules.
* **Discovery-only behavior:** Network Discovery must not infer, correlate, or interpret beyond its producing contract.

## 10. Governed Classification Field

### 10.1 Field Name

`service_hint`

### 10.2 Definition

`service_hint` is the Network Discovery Domain’s governed evidence field for static generic TCP port classification.

It represents the Network-owned static service label associated with a TCP port after successful connect-only TCP establishment.

It is derived strictly from:

* the destination TCP port
* successful TCP connection establishment
* the static Network port-to-service mapping defined in this specification

### 10.3 Constraints

`service_hint` MUST:

* be emitted only by Network Discovery for Network-owned findings
* be derived solely from the approved Network static mapping
* remain deterministic for identical input conditions
* use only allowed values defined by this specification
* remain producer-owned by Network Discovery
* represent generic port classification only

`service_hint` MUST NOT:

* encode protocol correctness
* imply service health
* imply authentication state
* imply TLS state
* imply certificate validity
* imply product identity
* imply application behavior
* imply telemetry ingestion health
* imply identity role
* imply host posture
* imply successful protocol negotiation
* be derived from response content, banners, payloads, handshakes, metadata, authentication results, or enumeration
* be normalized, aliased, extended, or silently remapped by consuming components unless explicitly authorized by an approved consuming specification

### 10.4 Allowed Values

The following values are authorized for Network `service_hint`:

* `FTP`
* `SSH`
* `DNS`
* `HTTP`
* `HTTPS`
* `Kerberos`
* `RPC`
* `NetBIOS-SSN`
* `LDAP`
* `SMB`
* `LDAPS`
* `RDP`
* `WinRM`
* `WinRM-HTTPS`
* `HTTP-Alt`
* `HTTPS-Alt`
* `SYSLOG`
* `SYSLOG-TLS`
* `GELF`
* `LOG-FORWARD`
* `OTLP-GRPC`
* `OTLP-HTTP`

No additional `service_hint` values are authorized without controlled specification update.

### 10.5 Cross-Domain Contract Requirements

Network Discovery is the authoritative producing owner of `service_hint` for Network findings.

Specialized domains MUST NOT reuse `service_hint` where their governing specifications define separate producer-owned fields, such as:

* `application_service_hint`
* `telemetry_logging_hint`

Where a consuming domain is authorized to consume `service_hint`, it MUST treat the field as:

* read-only
* authoritative within the Network producing contract
* exact-match only
* non-derivable outside Network Discovery
* non-normalized unless explicitly authorized

Consuming domains MUST NOT assume that a Network `service_hint` confirms that the expected protocol is correctly implemented or available beyond TCP connection success.

### 10.6 Determinism Requirements

For identical inputs and environmental conditions:

* the same TCP port MUST map to the same `service_hint`
* the static mapping MUST NOT vary at runtime
* no response-dependent classification is permitted
* no adaptive classification is permitted
* no heuristic remapping is permitted

## 11. Finding Model

All Network Discovery findings MUST conform to the `DiscoveryFinding` schema.

### 11.1 `host_presence`

#### Domain

`network`

#### Category

`host_presence`

#### Definition

Represents a host presence observation from the Network Discovery vantage point.

#### Required Evidence Fields

* `method`
* `reachable`
* `timeout_s`, where applicable

#### Constraints

A `host_presence` finding:

* reflects only the approved host presence observation method
* MUST NOT imply TCP service availability
* MUST NOT imply generalized host accessibility across all protocols
* MUST NOT suppress or invalidate TCP `open_port` findings
* MUST NOT be combined with TCP signals inside Network Discovery to infer host posture

### 11.2 `open_port`

#### Domain

`network`

#### Category

`open_port`

#### Definition

Represents successful connect-only TCP establishment to an approved TCP port from the Network Discovery vantage point.

#### Required Evidence Fields

* `method`
* `port`
* `service_hint`
* `timeout_s`

#### Required Evidence Values

* `method` SHOULD use the approved Network method value for TCP connect-only observations.
* `port` MUST be one of the approved TCP ports in Section 8.
* `service_hint` MUST be one of the approved values in Section 10.4.
* `timeout_s` MUST reflect the static timeout used for the TCP connection attempt.

#### Confidence

`open_port` confidence reflects confidence in the TCP connection observation and static port mapping only.

It MUST NOT be used to imply service health, protocol correctness, authentication state, TLS state, application behavior, identity role, telemetry ingestion state, product identity, or host posture.

### 11.3 `port_probe_error`

#### Domain

`network`

#### Category

`port_probe_error`

#### Definition

Represents a retained Network-owned probe issue where the implementation preserves structured diagnostic output for a TCP port attempt.

This category is permitted only as a Network-owned operational diagnostic finding. It is not an exposure finding.

#### Required Evidence Fields, If Retained

* `method`
* `port`
* `service_hint`
* `timeout_s`
* `error`

#### Constraints

A `port_probe_error` finding:

* MUST NOT be interpreted as an open port
* MUST NOT imply service exposure
* MUST NOT imply service absence
* MUST NOT imply host posture
* MUST NOT trigger specialized-domain semantic meaning
* MUST remain deterministic and non-sensitive
* MUST preserve Network-owned evidence semantics

Normal non-success outcomes may be represented as no finding or as retained structured diagnostics only where approved by implementation design and validation artifacts.

## 12. Discovery Overlap Rules

Network Discovery may produce `network/open_port` findings on ports also observed by specialized discovery domains.

Such overlap is permitted only when all of the following remain true:

* Network Discovery produces only Network-owned generic findings.
* Specialized domains produce only their own domain-specific findings.
* Each domain uses only its approved bounded observation model.
* Each finding preserves its own domain, category, evidence fields, controlled vocabulary, and semantic ownership.
* No domain suppresses another domain’s valid direct observation.
* No domain mutates, relabels, normalizes, aliases, or reinterprets another domain’s findings.
* No overlap is used to perform inference, correlation, protocol validation, service health assessment, response interpretation, or semantic reconciliation.

Examples of valid overlapping observations include:

* TCP/8080 may produce both:
  * `network/open_port` with `service_hint = "HTTP-Alt"`
  * `application_service/application_service_exposed` with the Application / Service governed classification field
* TCP/4317 may produce both:
  * `network/open_port` with `service_hint = "OTLP-GRPC"`
  * `telemetry_logging/telemetry_logging_exposed` with the Telemetry / Logging governed classification field
* TCP/445 may produce both:
  * `network/open_port` with `service_hint = "SMB"`
  * `identity/identity_service_exposed` with Identity-owned exposure semantics

These findings are not duplicates. They are independently valid producer-owned observations.

## 13. Determinism Requirements

Network Discovery MUST preserve deterministic behavior across all execution paths.

The following requirements apply:

* TCP probe order MUST be fixed.
* Host presence observation order relative to TCP probing MUST be fixed.
* Static port-to-service mapping MUST be fixed.
* Timeout values MUST be fixed unless explicitly configured through an approved mechanism.
* Findings MUST be emitted in deterministic order.
* Error handling MUST be deterministic and structured.
* No randomization is permitted.
* No concurrency is permitted unless separately authorized and proven deterministic.
* No adaptive probing is permitted.
* No response-dependent branching is permitted.
* Identical inputs and environmental conditions MUST produce identical outputs.

---

## 14. Failure and Error Handling

Network Discovery MUST handle errors gracefully.

Failure or non-success for one port MUST NOT prevent probing of subsequent approved ports.

Network Discovery MUST NOT allow unhandled exceptions to propagate in a way that terminates orchestration.

Error handling MUST:

* be structured
* be deterministic
* avoid sensitive data collection
* avoid response parsing
* avoid adaptive retry behavior
* preserve failure isolation
* support auditability

Connection refusal, timeout, and other bounded connection failures MUST NOT be interpreted as exposure.

A failure to observe one signal MUST NOT invalidate other independently observed signals.

## 19. Risk Assessment

| Risk | Impact | Mitigation |
|---|---:|---|
| Network absorbs specialized semantic ownership | High | Define `network/open_port` as generic TCP exposure only and preserve specialized-domain ownership |
| Specialized domains emit generic `open_port` findings | High | Preserve category ownership and prohibit specialized domains from emitting Network categories |
| Output / Reporting becomes semantic reconciliation layer | High | Preserve presentation-only behavior and structured findings as source of truth |
| Overlapping findings are treated as duplicates | High | Require producer-owned domain, category, evidence, and semantic separation |
| Expanded port inventory increases probe footprint | Medium | Maintain bounded approved inventory, fixed order, and low static timeouts |
| Protocol interaction drift | High | Enforce connect-only behavior and prohibit payloads, response reads, negotiation, TLS handshakes, and banners |
| Non-deterministic output | High | Enforce fixed port order, fixed mappings, and deterministic error handling |
| Hidden coupling with Host Configuration | High | Require exact-match, specification-bound downstream consumption only where authorized |
| Misinterpretation of `service_hint` | Medium | Define `service_hint` as static Network-owned classification only |
| Documentation cross-reference drift | Medium | Plan controlled post-approval reconciliation across impacted `03.xx`, `02.xx`, and validation artifacts |
