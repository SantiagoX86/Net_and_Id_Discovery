## 1. Purpose

This document defines the scope, constraints, and implementation requirements for the Host Configuration Discovery Domain (M6).  

The objective of this domain is to derive **externally observable host configuration posture signals** using strictly agentless and non-exploitative techniques.  

This domain introduces an interpretation layer that converts previously observed network and identity signals into structured host-level findings.

---

## 2. Scope

### 2.1 In Scope

The Host Configuration Discovery Domain is limited to **passive and connect-only inference** based on externally observable behavior.

The following signal types are in scope:

- Firewall and filtering behavior
  - Open vs closed vs filtered port characteristics
- Service exposure patterns
  - Combinations of exposed services indicating potential host roles
- Host reachability characteristics
  - ICMP responsiveness
  - Connection stability and timing consistency
- Port response behavior
  - Timeout vs connection refused patterns

All signals must be derived from:
- Existing TCP connect results
- Existing ICMP reachability checks

No new probing techniques are introduced.

---

### 2.2 Out of Scope

The following activities are strictly prohibited:

- Banner grabbing
- Protocol negotiation
- Authentication attempts
- OS fingerprinting via crafted packets
- Service enumeration (LDAP, SMB, RPC, etc.)
- Any exploitative or intrusive technique

This domain must remain:
- Agentless
- Non-exploitative
- Connect-only / passive

---

## 3. Architectural Constraints

The Host Configuration Discovery Domain must comply with all existing system constraints:

- No direct network interaction beyond existing probe outputs
- No modification of Network or Identity domain behavior
- No orchestration control logic
- Strict domain isolation

This domain may consume data only via:
- Normalized DiscoveryFinding objects
- Orchestrator-provided context

Direct inter-domain coupling is prohibited.

---

## 4. Discovery Model

### 4.1 Interaction Type

This domain does not perform direct probing.

All findings are derived from:
- TCP connect results
- ICMP reachability observations
- Previously generated findings

---

### 4.2 Core Principle

This domain introduces **interpretation, not detection**.

It must:
- Reuse existing signals
- Produce deterministic inferences
- Avoid introducing new network behavior

---

## 5. Finding Categories

The following categories are introduced under the `host_config` domain:

### 5.1 host_firewall_behavior
Describes observed filtering characteristics of the host based on port response patterns.

### 5.2 host_service_pattern
Describes structured interpretation of exposed service combinations.

### 5.3 host_posture_inferred
Provides high-level inference about host configuration or likely role based on observed signals.

---

## 6. Finding Structure

All findings must conform to the standard DiscoveryFinding schema.

### Example:

```json
{
  "domain": "host_config",
  "category": "host_firewall_behavior",
  "target": "192.168.1.10",
  "evidence": {
    "pattern": "filtered_ports_majority",
    "observed_behavior": ["timeout", "timeout", "refused"],
    "note": "Behavior consistent with host-based firewall filtering"
  },
  "confidence": 0.6,
  "observed_at": "ISO-8601 timestamp"
}
