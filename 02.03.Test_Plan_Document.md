## 1. Purpose

This Test Plan defines the approach, scope, and validation strategy for verifying the functionality, determinism, and security compliance of the Enterprise Security Discovery Orchestration Framework.

---

## 2. Scope

Testing will validate:

- Core Framework behavior
- Discovery domain functionality
- Inference-layer functionality
- Control-layer functionality
- Output & Reporting module accuracy
- Architectural boundary enforcement across discovery, inference, orchestration, control, and reporting layers
- Producer / consumer contract compliance
- Governed evidence field and controlled vocabulary handling
- Deterministic execution under controlled conditions

---

## 3. Test Environment

Testing will be conducted in a controlled lab environment:

- Source: Kali Linux VM (vmSCS)
- Target: Windows 10 VM
- Network: Isolated virtual network

No production systems are used.

---

## 4. Test Objectives

- Validate correct detection of network services via TCP connectivity
- Validate detection of identity-related service exposure
- Validate inference-layer behavior using authorized upstream findings only
- Confirm adherence to non-exploitative constraints
- Verify deterministic behavior across repeated runs
- Ensure output accuracy and completeness
- Validate architectural boundary enforcement across discovery, inference, orchestration, control, and reporting layers
- Validate producer / consumer contract compliance
- Validate governed evidence field and controlled vocabulary integrity

---

## 5. Test Scenarios

### 5.1 Network Discovery

- Open port detection (e.g., 80, 443, 22)
- Closed port handling
- Timeout behavior
- Discovery-only boundary validation

---

### 5.2 Identity Discovery

- Detection of identity service ports:
  - 88 (Kerberos)
  - 135 (RPC)
  - 139 (NetBIOS-SSN)
  - 389 (LDAP)
  - 445 (SMB)
  - 636 (LDAPS)
  - 3389 (RDP)
  - 5985 (WinRM)
  - 5986 (WinRM HTTPS)

- Validation of connect-only behavior
- No authentication or protocol negotiation
- Discovery-only boundary validation

---

### 5.3 Inference Layer / Host Configuration Validation

- Validation of derived findings using authorized upstream governed findings
- Read-only upstream finding handling
- Append-only derived output generation
- Contract-bound governed evidence handling
- No direct network interaction by inference logic

---

### 5.4 Core / Control / Output Layer Validation

- Core orchestration-only behavior
- Control-layer input validation and invocation behavior
- Output presentation-only behavior
- No hidden coupling across layers

---

### 5.5 Governed Evidence Field / Controlled Vocabulary Validation

- Canonical governed evidence value validation
- Exact-match consumer handling validation
- No reinterpretation of governed evidence fields
- Cross-artifact controlled vocabulary consistency

---

### 5.6 Output Validation

- JSON output correctness
- Markdown report formatting
- Evidence and timestamp accuracy

---

## 6. Success Criteria

Testing is successful when:

- All planned validation areas execute without error
- Findings are accurate and properly structured
- No security constraints are violated
- Output is deterministic across repeated runs
- Discovery domains are validated as discovery-only and emit no inference or cross-domain correlation
- Inference-layer logic is validated to consume upstream findings as read-only, append-only, and contract-bound
- Core, Control, and Output layers are validated as free of domain-specific business logic
- Governed evidence fields are validated as exact-match, deterministic, and cross-artifact consistent

---

## 7. Risk Considerations

- Environmental variability affecting results
- Network instability impacting connectivity tests
- Architectural boundary drift between discovery, inference, control, orchestration, and reporting layers
- Hidden coupling between producing and consuming components
- Governed evidence field or controlled vocabulary drift across specifications, implementation, and test artifacts
- Business logic leakage into non-domain layers

Mitigation:
- Controlled lab setup
- Consistent VM configurations
- Validation against approved architectural and domain-isolation controls
- Cross-artifact reconciliation of governed contracts and evidence vocabulary
- Explicit boundary enforcement testing across all relevant layers
